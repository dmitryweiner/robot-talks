Q: Вот есть разные парадигмы программирования. Например, в императивном стиле автор контролирует порядок выполнения и память. В ООП только порядок выполнения, память как-то сама выделяется (с помощью new), с поинтерами не надо возиться. В функциональном программировании и порядок выполнения, и память и отдаются на откуп компилятору. А какие ещё подходы есть? Полностью декларативный стиль?

A: Да, можно смотреть на парадигмы именно как на **степень передачи контроля от программиста к системе**. Но я бы чуть поправил шкалу:

императивное программирование — я сам говорю, **как делать**;  
 декларативное — я описываю, **что должно получиться**;  
 система сама выбирает конкретный способ выполнения.

Но “память” и “порядок выполнения” — это только две оси. Есть ещё управление состоянием, ошибками, временем, параллелизмом, вводом-выводом, поиском решения, жизненным циклом объектов и так далее.

## **1\. Императивное программирование**

Это базовый стиль:

```
let sum = 0;

for (const x of numbers) {
 sum += x;
}
```

Ты прямо говоришь машине:

1. заведи переменную;  
2. пройди по массиву;  
3. каждый раз измени `sum`;  
4. в конце получишь результат.

Здесь программист контролирует **порядок действий** и **изменение состояния**.

C — почти чистый пример: ты контролируешь и порядок, и память, и указатели, и время жизни объектов.

JavaScript, Python, Java — тоже в основном императивные языки, но память уже автоматизирована сборщиком мусора.

## **2\. Структурное программирование**

Это не совсем отдельная большая парадигма, а скорее важная историческая ступень: “давайте перестанем прыгать по коду через `goto`, а будем строить программы из блоков”.

То есть:

```
последовательность
условие
цикл
функция
```

Вместо хаотичного потока управления:

```
прыгай сюда
прыгай туда
если что — перепрыгни в середину другой процедуры
```

Структурное программирование — это момент, когда люди поняли: **контроль над порядком выполнения нужен, но его надо дисциплинировать**.

## **3\. Процедурное программирование**

Это императивный стиль, организованный вокруг процедур/функций:

```
open_file();
read_data();
process_data();
write_result();
close_file();
```

Главная идея: программа — это набор процедур, которые вызывают друг друга.

Данные обычно отдельно, процедуры отдельно.

Это стиль C, Pascal, раннего Fortran, shell-скриптов и большого количества “обычного” кода.

## **4\. Объектно-ориентированное программирование**

ООП передаёт часть контроля не компилятору, а **объектной модели**.

Ты уже не просто вызываешь процедуру:

```
drawCircle(canvas, circle);
```

А говоришь:

```
circle.draw(canvas);
```

Идея в том, что данные и поведение живут вместе:

```
class Door {
 open() {
   this.isOpen = true;
 }

 close() {
   this.isOpen = false;
 }
}
```

ООП хорошо подходит, когда в программе есть много “вещей” с состоянием и поведением: окна, кнопки, документы, соединения, персонажи, заказы, устройства.

Но важный момент: ООП не обязательно означает “память сама”. В C++ есть ООП, но память может быть ручной. А в JavaScript есть сборщик мусора, но можно писать почти без классов.

То есть автоматическая память — это не свойство ООП как такового, а свойство конкретной runtime-среды.

## **5\. Функциональное программирование**

Функциональный стиль пытается уменьшить роль изменяемого состояния.

Императивно:

```
let result = [];

for (const user of users) {
 if (user.active) {
   result.push(user.name);
 }
}
```

Функционально:

```
const result = users
 .filter(user => user.active)
 .map(user => user.name);
```

Ты уже меньше говоришь “как именно шагать по массиву”, а больше описываешь преобразование:

```
возьми пользователей
оставь активных
преврати каждого в имя
```

В сильном функциональном стиле программа выглядит как композиция функций:

```
данные → функция → функция → функция → результат
```

Порядок выполнения частично отдаётся системе, особенно в ленивых языках вроде Haskell. Компилятор может сам решать, когда реально вычислять значение.

Но функциональное программирование — не обязательно “всё отдаём компилятору”. Скорее:

программист отказывается от ручного управления изменяемым состоянием и описывает вычисления как преобразование значений.

## **6\. Декларативное программирование**

Вот это как раз стиль “я говорю, что хочу, а не как получить”.

SQL — классический пример:

```
SELECT name
FROM users
WHERE active = true
ORDER BY created_at DESC;
```

Ты не говоришь:

```
открой файл таблицы
пройди по строкам
проверь поле active
положи подходящие строки в буфер
отсортируй quicksort-ом
верни результат
```

Ты говоришь:

```
дай мне имена активных пользователей, отсортированных по дате
```

А база сама решает:

* использовать ли индекс;  
* в каком порядке читать таблицы;  
* какой алгоритм join выбрать;  
* когда сортировать;  
* как распараллелить запрос.

Это очень сильный пример “отдать контроль системе”.

## **7\. Логическое программирование**

Это ещё более декларативный стиль. Например Prolog.

Ты не пишешь алгоритм поиска ответа. Ты описываешь факты и правила:

```
parent(alice, bob).
parent(bob, charlie).

grandparent(X, Z) :-
 parent(X, Y),
 parent(Y, Z).
```

А потом спрашиваешь:

```
grandparent(alice, Who).
```

И система сама ищет `Who`.

Здесь программист отдаёт системе не только порядок выполнения, но и сам **поиск решения**.

Очень интересная парадигма: программа становится не инструкцией, а набором утверждений о мире.

## **8\. Constraint programming — программирование ограничениями**

Похоже на логическое, но акцент на ограничениях.

Например:

```
x + y = 10
x > 3
y < 8
```

А система должна подобрать значения `x` и `y`.

Это используется в задачах расписаний, планирования, оптимизации, компоновки, SAT/SMT-солверах.

Например: “расставь 20 встреч по комнатам так, чтобы никто не пересекался, у каждой встречи была подходящая комната, а важные люди не начинали раньше 10 утра”.

Ты не пишешь руками все перестановки. Ты описываешь ограничения.

Это ещё один уровень передачи контроля:

я не знаю путь к ответу; я знаю свойства правильного ответа.

## **9\. Реактивное программирование**

Здесь программа описывается как сеть зависимостей и реакций на изменения.

Например:

```
const fullName = computed(() => firstName.value + " " + lastName.value);
```

Ты не говоришь:

```
каждый раз после изменения firstName не забудь пересчитать fullName
каждый раз после изменения lastName тоже не забудь пересчитать fullName
```

Ты описываешь зависимость:

```
fullName зависит от firstName и lastName
```

А система сама пересчитывает.

Это стиль React, Vue, Solid, RxJS, spreadsheet-like systems.

Кстати, электронная таблица — один из лучших бытовых примеров реактивного декларативного программирования:

```
C1 = A1 + B1
```

Потом меняется `A1`, и `C1` пересчитывается сам.

## **10\. Dataflow programming — поток данных**

Программа описывается как граф:

```
данные → фильтр → преобразование → агрегация → вывод
```

Узлы графа — операции, рёбра — потоки данных.

Примеры:

* Node-RED;  
* LabVIEW;  
* Unix pipes;  
* Apache Beam;  
* TensorFlow-графы в старом стиле;  
* audio/video processing graphs;  
* визуальные редакторы shader nodes.

Например в shell:

```
cat access.log | grep 404 | sort | uniq -c
```

Ты строишь конвейер. Данные текут через стадии.

Это не совсем “полностью декларативно”, но управление переходит от “сделай шаг 1, потом шаг 2” к “вот граф обработки”.

## **11\. Event-driven programming — событийное программирование**

Программа строится вокруг событий:

```
button.addEventListener("click", () => {
 console.log("clicked");
});
```

Ты не контролируешь главный цикл напрямую. Он где\-то внутри браузера, ОС или framework-а.

Ты говоришь:

```
когда случится вот это событие — вызови этот код
```

Это стиль GUI, браузера, Node.js, игр, серверов, embedded-систем.

Контроль инвертирован:

не ты вызываешь систему, а система вызывает тебя.

Это называется **inversion of control**.

## **12\. Actor model — акторная модель**

В акторной модели программа состоит из независимых “акторов”, которые обмениваются сообщениями.

```
actor A → message → actor B
actor B → message → actor C
```

Каждый актор имеет своё состояние, но не делит его напрямую с другими.

Это подход Erlang, Elixir, Akka.

Очень хорош для распределённых и отказоустойчивых систем. Вместо общей памяти и локов:

```
не трогай мои данные;
пришли мне сообщение;
я сам решу, что с ним сделать.
```

Тут программист отдаёт часть контроля планировщику акторов: кто когда выполнится, в каком порядке придут сообщения, как распределить работу по процессам.

## **13\. Concurrent / parallel programming**

Это не одна парадигма, а семейство подходов.

Есть разные модели:

```
threads + locks
async/await
actors
CSP/channels
STM
GPU kernels
map-reduce
```

Например CSP, как в Go:

```
messages <- "hello"
msg := <-messages
```

Процессы общаются через каналы.

Идея:

не разделяй память, а передавай сообщения.

Это способ контролировать параллелизм без полного ручного управления потоками и mutex-ами.

## **14\. Array programming**

Очень красивая парадигма из APL/J/K, NumPy, MATLAB.

Ты работаешь не с отдельными числами, а с целыми массивами как с едиными значениями.

Вместо:

```
for (let i = 0; i < xs.length; i++) {
 ys[i] = xs[i] * 2;
}
```

Пишешь концептуально:

```
ys = xs * 2
```

И это применяется ко всему массиву.

В NumPy:

```
distances = np.sqrt(xs ** 2 + ys ** 2)
```

Ты описываешь формулу для массивов, а система решает, как эффективно это выполнить.

Это близко к функциональному и декларативному стилю, но с особым мышлением: **операции над целыми формами данных**.

## **15\. Generic programming**

Это программирование через обобщённые структуры и алгоритмы.

Например в TypeScript:

```
function first<T>(items: T[]): T | undefined {
 return items[0];
}
```

Или в C++ templates.

Ты пишешь алгоритм не для конкретного типа, а для класса типов:

```
это работает для любого T, если T удовлетворяет условиям
```

В более развитой форме это превращается в typeclasses, traits, protocols.

Например в Rust:

```
fn print<T: Display>(value: T) {
   println!("{}", value);
}
```

Ты говоришь:

```
мне не важно, что это за тип;
важно, чтобы его можно было показать как текст.
```

Это парадигма, где часть логики переносится в **систему типов**.

## **16\. Metaprogramming — метапрограммирование**

Это когда программа пишет или преобразует программу.

Примеры:

* макросы Lisp;  
* template metaprogramming в C++;  
* decorators в Python;  
* code generation;  
* Babel/TypeScript transforms;  
* Rust macros;  
* compile-time evaluation.

Например:

```
#[derive(Debug, Clone)]
struct User {
   name: String,
}
```

Ты не пишешь вручную реализацию `Debug` и `Clone`; компилятор генерирует её.

Здесь ты передаёшь часть работы не runtime-системе, а **этапу компиляции**.

## **17\. Aspect-oriented programming**

Менее популярная, но интересная парадигма.

Идея: есть сквозные аспекты, которые проходят через много мест программы:

* логирование;  
* авторизация;  
* транзакции;  
* метрики;  
* кеширование;  
* обработка ошибок.

Вместо того чтобы писать в каждом методе:

```
logStart();
checkPermissions();
try {
 doWork();
} finally {
 logEnd();
}
```

Ты описываешь аспект:

```
перед каждым вызовом методов такого типа проверяй права
после каждого вызова логируй результат
```

Это мощно, но опасно: поведение программы становится менее очевидным.

## **18\. Declarative UI**

Это отдельная современная форма декларативного программирования.

Старый императивный UI:

```
const button = document.createElement("button");
button.textContent = "Save";
button.disabled = !formIsValid;
container.appendChild(button);
```

Декларативный UI:

```
<Button disabled={!formIsValid}>
 Save
</Button>
```

Ты описываешь, **как должен выглядеть интерфейс при данном состоянии**, а React/Vue/Solid решают, какие реальные изменения внести в DOM.

Это очень важный сдвиг:

UI \= функция от состояния.

```
state → view
```

Не “найди кнопку и поменяй ей текст”, а “при таком состоянии интерфейс должен быть таким”.

## **19\. Rule-based programming**

Программа состоит из правил вида:

```
если условие — выполни действие
```

Например:

```
если температура > 80°C, включить вентилятор
если пользователь VIP и корзина > 500, дать скидку
если существо голодно и видит еду, идти к еде
```

Это часто используется в экспертных системах, игровых AI, бизнес-правилах, fraud detection, автоматизации.

От обычного `if` отличается тем, что правила могут храниться отдельно от программы, комбинироваться, включаться/выключаться, исполняться движком.

## **20\. Probabilistic programming**

Ты описываешь вероятностную модель, а система делает вывод.

Примерно:

```
вероятность болезни зависит от симптомов
симптомы наблюдаются с шумом
найди наиболее вероятную болезнь
```

Псевдокод:

```
disease ~ Categorical(...)
fever ~ Bernoulli(probability_given(disease))
cough ~ Bernoulli(probability_given(disease))
observe(fever=True)
observe(cough=True)
infer(disease)
```

Здесь программист не пишет алгоритм Байесовского вывода вручную. Он описывает модель, а система сама считает распределения.

Примеры: Stan, PyMC, Edward, Turing.jl.

Это прямой родственник декларативного стиля:

я описываю вероятностный мир; система вычисляет, что вероятно при наблюдениях.

## **21\. Differentiable programming**

Это стиль, где программа строится так, чтобы по ней можно было автоматически считать производные.

Примеры: PyTorch, JAX, TensorFlow.

Ты пишешь вычисление:

```
loss = model(x).distance_to(y)
```

А система сама строит производные:

```
loss.backward()
```

Это фундаментально для нейросетей.

Тут программист отдаёт системе контроль над сложнейшей частью: вычислением градиентов.

## **22\. Genetic / evolutionary programming**

Здесь программа не столько пишется, сколько **выращивается**.

Задаётся:

* пространство возможных программ;  
* критерий отбора;  
* мутации;  
* скрещивание;  
* fitness-функция.

Система генерирует варианты, проверяет их, отбирает лучшие.

Это уже почти противоположность ручному программированию:

программист пишет не решение, а среду, в которой решение может эволюционировать.

Примерно:

```
создай 1000 случайных стратегий
оцени каждую
оставь лучшие
мутируй
повтори
```

Используется в оптимизации, генерации стратегий, дизайне антенн, игровых агентов, поиске архитектур.

## **23\. Intentional / model-driven programming**

Здесь программист описывает модель предметной области, а код генерируется из неё.

Например:

```
Entity: User
fields:
 name: string
 email: string unique
 createdAt: datetime
```

А система генерирует:

* SQL-таблицы;  
* API;  
* формы;  
* валидацию;  
* типы;  
* документацию.

Это часто встречается в low-code, DSL, ORM, backend-фреймворках.

Ты пишешь не программу, а **модель**, из которой программа выводится.

## **24\. DSL-oriented programming**

DSL — domain-specific language, язык для конкретной области.

Например:

* SQL для запросов;  
* HTML для структуры документа;  
* CSS для стилей;  
* Regex для поиска текста;  
* GraphQL для API-запросов;  
* shader language для графики;  
* Terraform для инфраструктуры;  
* Kubernetes YAML для развертывания.

DSL часто декларативны.

Например CSS:

```
.button {
 color: red;
 padding: 8px;
}
```

Ты не говоришь браузеру, как именно рисовать пиксели. Ты описываешь свойства элементов.

HTML/CSS — мощный пример почти полностью декларативной системы.

## **Если собрать всё в одну шкалу**

Можно примерно так:

```
C/Assembly
 я контролирую почти всё

Imperative / Procedural
 я контролирую порядок действий

OOP
 я моделирую мир объектами, система управляет памятью и диспетчеризацией

Functional
 я описываю преобразования значений, меньше управляю состоянием

Reactive / Dataflow
 я описываю зависимости, система сама обновляет результат

SQL / Declarative UI / CSS
 я описываю желаемый результат

Logic / Constraint / Probabilistic
 я описываю правила, ограничения или модель мира; система ищет ответ

Evolutionary / ML / Differentiable
 я описываю критерий, данные или среду; система подбирает решение
```

## **Самая интересная граница**

Есть важный переход:

```
1. Написать алгоритм
2. Описать результат
3. Описать ограничения на результат
4. Описать критерий качества результата
5. Описать среду, где результат сам появится
```

Например:

### **Императивно**

```
вот точные шаги, как построить расписание
```

### **Декларативно**

```
вот каким должно быть расписание
```

### **Constraint programming**

```
вот ограничения, которым расписание должно удовлетворять
```

### **Optimization / evolutionary**

```
вот что значит “хорошее расписание”, найди лучшее
```

### **Machine learning**

```
вот примеры хороших расписаний, научись сам
```

Это очень красивая лестница: чем выше, тем меньше ты пишешь конкретное поведение, и тем больше задаёшь **пространство возможных поведений**.

## **“Полностью декларативный стиль” существует?**

Да, но с оговорками.

SQL, HTML, CSS, Prolog, Terraform, Nix, Kubernetes-манифесты, регулярные выражения, constraint solvers — всё это очень декларативно.

Но полностью декларативная программа в реальном мире часто всё равно где\-то упирается в императивное ядро:

```
описание → планировщик/движок → реальные действия
```

Например:

* SQL-запрос исполняет императивный движок базы;  
* CSS применяет императивный layout/rendering engine браузера;  
* React JSX превращается в реальные операции над DOM;  
* Terraform-план превращается в API-вызовы;  
* Prolog всё равно делает поиск по определённой стратегии.

То есть декларативность обычно означает не “императивности нет вообще”, а:

императивность спрятана внутрь движка, а пользователь работает на более высоком уровне описания.

И, пожалуй, это главный паттерн всей истории программирования:

```
то, что раньше писали руками,
становится внутренностью платформы.
```

Сначала вручную управляли регистрами.  
 Потом переменными.  
 Потом объектами.  
 Потом зависимостями.  
 Потом ограничениями.  
 Потом критериями обучения.

Программирование всё время двигается от:

```
сделай вот так
```

к:

```
пусть мир имеет вот такие свойства
```
