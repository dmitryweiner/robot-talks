# JavaScript: как не стрелять себе в ногу на уровне языковых конструкций

## Q: А есть какие-нибудь примеры, как не надо писать в JavaScript, именно на уровне языковых конструкции, когда неумышленно стреляешь себе в ногу. Ну типа по привычке пишешь `const [result] = getResult() || [];` приходит 0 и программа крашится в рантайме (а написали бы `?? []` всё бы отработало). Можно ещё побольше таких примеров и как с этим бороться?

## A:

Да. В JavaScript есть целый класс багов, которые выглядят как «ну я же нормально написал», а потом выясняется, что языковая конструкция означает не совсем то, что мозг ожидал.

Ниже — набор типичных языковых мин: falsy/nullish, destructuring, `this`, coercion, async, equality, optional chaining, мутирующие методы и похожие ловушки.

---

# 1. `||` вместо `??` для дефолтов

Плохо:

```js
const [result] = getResult() || [];
```

Если `getResult()` вернёт `0`, получится:

```js
const [result] = 0;
// TypeError: 0 is not iterable
```

Проблема: `||` заменяет любое falsy-значение:

```js
false
0
""
NaN
null
undefined
```

А часто мы хотим подставлять дефолт только при `null` или `undefined`.

Лучше:

```js
const [result] = getResult() ?? [];
```

Но если функция теоретически может вернуть `0`, а ты потом делаешь array destructuring, то сама модель типов подозрительная. Ещё лучше:

```js
const value = getResult();
const [result] = Array.isArray(value) ? value : [];
```

Как бороться:

```json
{
  "rules": {
    "@typescript-eslint/prefer-nullish-coalescing": "warn"
  }
}
```

Правило в голове:

```js
value || defaultValue  // плохо для чисел, строк, boolean
value ?? defaultValue  // хорошо, если default нужен только для null/undefined
```

---

# 2. Дефолт в параметрах работает только для `undefined`, но не для `null`

Плохо:

```js
function renderTitle(title = "Untitled") {
  return title.toUpperCase();
}

renderTitle(undefined); // "UNTITLED"
renderTitle(null);      // TypeError
```

Default parameter применяется только когда аргумент равен `undefined`.

Лучше:

```js
function renderTitle(title) {
  const safeTitle = title ?? "Untitled";
  return safeTitle.toUpperCase();
}
```

---

# 3. Destructuring из `undefined` / `null`

Плохо:

```js
function renderUser({ name }) {
  console.log(name);
}

renderUser(undefined);
// TypeError: Cannot destructure property 'name' of undefined
```

Частично лучше:

```js
function renderUser(user = {}) {
  const { name } = user;
}
```

Но это не спасает от `null`.

Надёжнее:

```js
function renderUser(user) {
  const { name } = user ?? {};
}
```

Вложенный destructuring тоже опасен:

```js
const {
  settings: {
    theme
  }
} = user;
```

Упадёт, если нет `settings`.

Лучше:

```js
const theme = user?.settings?.theme ?? "light";
```

---

# 4. Optional chaining не распространяется дальше, чем кажется

Безопасно:

```js
user?.profile?.name
```

Опасно:

```js
(user?.profile).name
```

Если `user` равен `null`, выражение `user?.profile` вернёт `undefined`, а потом `.name` будет читаться уже у `undefined`.

Лучше:

```js
const name = user?.profile?.name;
```

Ещё тонкий случай:

```js
user?.getName();
```

Это безопасно, если `user` отсутствует. Но если `user` есть, а `getName` не функция — упадёт.

Если сама функция опциональна:

```js
user.getName?.();
```

Если опциональны оба:

```js
user?.getName?.();
```

---

# 5. `array.filter(Boolean)` случайно удаляет валидные значения

Плохо:

```js
const values = [0, 1, 2, null, undefined, 3];

const clean = values.filter(Boolean);
// [1, 2, 3]
```

`0` потерялся. Иногда это баг.

То же со строками:

```js
["hello", "", "world", null].filter(Boolean);
// ["hello", "world"]
```

Если пустая строка — валидное значение, ты её случайно удалил.

Лучше явно:

```js
const clean = values.filter(value => value != null);
```

Это уберёт только `null` и `undefined`.

В TypeScript можно сделать type guard:

```ts
function isPresent<T>(value: T | null | undefined): value is T {
  return value !== null && value !== undefined;
}

const clean = values.filter(isPresent);
```

---

# 6. `map(parseInt)`

Плохо:

```js
["10", "10", "10"].map(parseInt);
// [10, NaN, 2]
```

Почему? `map` передаёт в callback три аргумента:

```js
(value, index, array)
```

А `parseInt` принимает второй аргумент как radix:

```js
parseInt("10", 0); // 10
parseInt("10", 1); // NaN
parseInt("10", 2); // 2
```

Лучше:

```js
["10", "10", "10"].map(value => parseInt(value, 10));
```

Или:

```js
["10", "10", "10"].map(Number);
// [10, 10, 10]
```

Но `Number` и `parseInt` ведут себя по-разному:

```js
Number("10px");    // NaN
parseInt("10px");  // 10
```

---

# 7. `for...in` по массиву вместо `for...of`

Плохо:

```js
const arr = ["a", "b", "c"];

for (const item in arr) {
  console.log(item);
}
```

Выведет:

```js
"0"
"1"
"2"
```

`for...in` проходит по ключам объекта, а не по значениям.

Лучше:

```js
for (const item of arr) {
  console.log(item);
}
```

Если нужны индексы:

```js
for (const [index, item] of arr.entries()) {
  console.log(index, item);
}
```

Правило:

```js
for...in  // ключи объекта
for...of  // значения iterable
```

---

# 8. `typeof null === "object"`

```js
typeof null;
// "object"
```

Плохо:

```js
function isObject(value) {
  return typeof value === "object";
}

isObject(null); // true
```

Лучше:

```js
function isObject(value) {
  return value !== null && typeof value === "object";
}
```

Если нужно отличить объект от массива:

```js
function isRecord(value) {
  return value !== null && typeof value === "object" && !Array.isArray(value);
}
```

---

# 9. `NaN` не равен самому себе

```js
NaN === NaN;
// false
```

Плохо:

```js
if (value === NaN) {
  // никогда не выполнится
}
```

Лучше:

```js
Number.isNaN(value)
```

Важно: глобальный `isNaN` делает coercion:

```js
isNaN("hello");        // true
Number.isNaN("hello"); // false
```

Обычно нужен именно `Number.isNaN`.

---

# 10. `isFinite` тоже делает coercion

Плохо:

```js
isFinite("123"); // true
```

Лучше:

```js
Number.isFinite("123"); // false
Number.isFinite(123);   // true
```

---

# 11. `+` как сложение и конкатенация

```js
1 + 2;     // 3
"1" + 2;   // "12"
1 + "2";   // "12"
```

Особенно мерзко в `reduce`:

```js
const total = values.reduce((sum, value) => sum + value);
```

Если где-то затесалась строка:

```js
[1, 2, "3", 4].reduce((sum, value) => sum + value);
// "334"
```

Лучше:

```js
const total = values.reduce((sum, value) => sum + Number(value), 0);
```

Или валидировать:

```js
const total = values.reduce((sum, value) => {
  if (typeof value !== "number") {
    throw new TypeError("Expected number");
  }

  return sum + value;
}, 0);
```

---

# 12. `reduce` без начального значения

Плохо:

```js
const sum = numbers.reduce((acc, x) => acc + x);
```

На пустом массиве:

```js
[].reduce((acc, x) => acc + x);
// TypeError: Reduce of empty array with no initial value
```

Лучше почти всегда писать initial value:

```js
const sum = numbers.reduce((acc, x) => acc + x, 0);
```

Для объектов:

```js
const byId = users.reduce((acc, user) => {
  acc[user.id] = user;
  return acc;
}, {});
```

В TypeScript лучше явно:

```ts
const byId = users.reduce<Record<string, User>>((acc, user) => {
  acc[user.id] = user;
  return acc;
}, {});
```

---

# 13. `sort()` сортирует строки, а не числа

Плохо:

```js
[1, 2, 10, 20].sort();
// [1, 10, 2, 20]
```

По умолчанию сортировка лексикографическая.

Лучше:

```js
[1, 2, 10, 20].sort((a, b) => a - b);
// [1, 2, 10, 20]
```

Но `sort()` мутирует массив:

```js
const original = [3, 1, 2];
const sorted = original.sort((a, b) => a - b);

console.log(original); // [1, 2, 3]
```

Если не хочешь мутировать:

```js
const sorted = [...original].sort((a, b) => a - b);
```

Или современно:

```js
const sorted = original.toSorted((a, b) => a - b);
```

---

# 14. Мутирующие методы массива

Опасные методы:

```js
sort()
reverse()
splice()
push()
pop()
shift()
unshift()
fill()
copyWithin()
```

Плохо:

```js
const reversed = items.reverse();
```

Ты думаешь, что получил новый массив, а на самом деле испортил старый.

Лучше:

```js
const reversed = [...items].reverse();
```

Или:

```js
const reversed = items.toReversed();
```

Для `splice`:

```js
const newItems = items.toSpliced(index, 1);
```

Для замены элемента:

```js
const newItems = items.with(index, newValue);
```

В React это особенно важно: мутация массива может привести к странным багам с ререндером.

---

# 15. `Array(n).map(...)` не работает как ожидается

Плохо:

```js
Array(5).map((_, i) => i);
// [empty × 5]
```

Массив содержит “дырки”, а `map` по ним не проходит.

Лучше:

```js
Array.from({ length: 5 }, (_, i) => i);
// [0, 1, 2, 3, 4]
```

Или:

```js
[...Array(5)].map((_, i) => i);
```

Но `Array.from` обычно понятнее.

---

# 16. Дырки в массивах ведут себя странно

```js
const arr = [1, , 3];

arr.length; // 3

arr.map(x => x * 2);
// [2, empty, 6]

arr.filter(Boolean);
// [1, 3]
```

А `for...of` выведет:

```js
1
undefined
3
```

Разные операции по-разному относятся к holes.

Лучше не создавать дырявые массивы. Если нужно пустое значение — класть явно:

```js
[1, undefined, 3]
```

---

# 17. `new Array(3)` vs `[3]`

```js
new Array(3);
// [empty × 3]

new Array("3");
// ["3"]
```

Это неприятный overloaded-конструктор.

Лучше почти никогда не использовать `new Array`.

Вместо:

```js
new Array(3).fill(0)
```

лучше:

```js
Array.from({ length: 3 }, () => 0)
```

---

# 18. `fill` с объектами создаёт одну и ту же ссылку

Плохо:

```js
const rows = Array(3).fill({ selected: false });

rows[0].selected = true;

console.log(rows);
// все три объекта стали selected: true
```

Потому что один объект разложили во все ячейки.

Лучше:

```js
const rows = Array.from({ length: 3 }, () => ({ selected: false }));
```

То же с матрицами:

```js
const matrix = Array(3).fill(Array(3).fill(0));
```

Все строки — одна и та же строка.

Лучше:

```js
const matrix = Array.from(
  { length: 3 },
  () => Array.from({ length: 3 }, () => 0)
);
```

---

# 19. `Date` мутируется

Плохо:

```js
const date = new Date("2026-01-01");

const next = date;
next.setDate(next.getDate() + 1);

console.log(date);
// тоже изменился
```

Лучше:

```js
const date = new Date("2026-01-01");

const next = new Date(date);
next.setDate(next.getDate() + 1);
```

---

# 20. `Date` парсит строки не всегда так, как хочется

```js
new Date("2026-01-01")
```

Это интерпретируется как дата в UTC, а не обязательно как локальная полночь.

А вот:

```js
new Date(2026, 0, 1)
```

Это локальное время. Месяцы при этом начинаются с нуля:

```js
new Date(2026, 1, 1);
// 1 февраля 2026
```

Для дат без времени лучше хранить строку:

```js
"2026-01-01"
```

А для времени — ISO с timezone:

```js
"2026-01-01T10:00:00+02:00"
```

---

# 21. `==` вместо `===`

Плохо:

```js
0 == false;        // true
"" == false;       // true
"0" == false;      // true
null == undefined; // true
[] == false;       // true
```

Почти всегда лучше:

```js
value === expected
```

Единственное популярное осознанное исключение:

```js
value == null
```

Это короткая проверка на `null` или `undefined`:

```js
value === null || value === undefined
```

ESLint:

```json
{
  "rules": {
    "eqeqeq": ["error", "always"]
  }
}
```

Или вариант, который разрешает `x == null`:

```json
{
  "rules": {
    "eqeqeq": ["error", "always", { "null": "ignore" }]
  }
}
```

---

# 22. `this` теряется при передаче метода

Плохо:

```js
const user = {
  name: "Alice",
  getName() {
    return this.name;
  }
};

const getName = user.getName;

getName(); // undefined или TypeError в strict mode
```

`this` определяется не местом объявления, а способом вызова.

Работает:

```js
user.getName();
```

Не работает:

```js
const fn = user.getName;
fn();
```

Лучше:

```js
const getName = user.getName.bind(user);
```

Или:

```js
const getName = () => user.getName();
```

---

# 23. Стрелочные функции не имеют своего `this`

Плохо:

```js
const user = {
  name: "Alice",
  getName: () => this.name
};

user.getName(); // не "Alice"
```

Стрелка берёт `this` из внешней области, а не из объекта.

Лучше:

```js
const user = {
  name: "Alice",
  getName() {
    return this.name;
  }
};
```

Простое правило:

```js
// метод объекта
getName() {}

// callback, closure, обработчик
() => {}
```

---

# 24. `arguments` в стрелочных функциях

Плохо:

```js
const fn = () => {
  console.log(arguments);
};
```

У стрелочной функции нет собственного `arguments`.

Лучше:

```js
const fn = (...args) => {
  console.log(args);
};
```

---

# 25. `return` и перенос строки

Плохо:

```js
function getUser() {
  return
  {
    name: "Alice"
  };
}
```

Функция вернёт `undefined` из-за automatic semicolon insertion.

Лучше:

```js
function getUser() {
  return {
    name: "Alice"
  };
}
```

То же с JSX:

```jsx
return
  <Component />;
```

Плохо.

Нужно:

```jsx
return (
  <Component />
);
```

---

# 26. `async` внутри `forEach`

Плохо:

```js
items.forEach(async item => {
  await save(item);
});

console.log("done");
```

`done` выведется до того, как `save` завершится. `forEach` не ждёт промисы.

Лучше последовательно:

```js
for (const item of items) {
  await save(item);
}

console.log("done");
```

Или параллельно:

```js
await Promise.all(items.map(item => save(item)));

console.log("done");
```

---

# 27. `try/catch` не ловит ошибку из промиса, которого не подождали (забыт `await`)

Плохо:

```js
try {
  doAsyncThing();
} catch (error) {
  console.error(error);
}
```

Если `doAsyncThing` возвращает rejected promise, `catch` его не поймает.

Нужно:

```js
try {
  await doAsyncThing();
} catch (error) {
  console.error(error);
}
```

Или:

```js
doAsyncThing().catch(console.error);
```

---

# 28. `Promise.all` падает целиком при первой ошибке

```js
await Promise.all([
  fetchUser(),
  fetchPosts(),
  fetchSettings()
]);
```

Если один промис зареджектится, весь `Promise.all` зареджектится.

Если нужны все результаты, даже с ошибками:

```js
const results = await Promise.allSettled([
  fetchUser(),
  fetchPosts(),
  fetchSettings()
]);
```

---

# 29. Забытый `await`

Плохо:

```js
const user = fetchUser();

console.log(user.name); // undefined
```

`user` — это Promise, а не пользователь.

Лучше:

```js
const user = await fetchUser();

console.log(user.name);
```

ESLint для TypeScript:

```json
{
  "rules": {
    "@typescript-eslint/no-floating-promises": "error"
  }
}
```

---

# 30. `await` внутри `map` без `Promise.all`

Плохо:

```js
const results = items.map(async item => {
  return await process(item);
});
```

`results` — это массив промисов, а не массив результатов.

Лучше:

```js
const results = await Promise.all(
  items.map(item => process(item))
);
```

---

# 31. `await` в цикле случайно делает всё последовательно

```js
const results = [];

for (const item of items) {
  results.push(await fetchItem(item));
}
```

Если запросы независимы, лучше:

```js
const results = await Promise.all(
  items.map(item => fetchItem(item))
);
```

Но если нужно соблюдать порядок эффектов или не перегрузить сервер — последовательный вариант нормальный.

---

# 32. Object spread делает только shallow copy

Плохо:

```js
const user = {
  name: "Alice",
  settings: {
    theme: "dark"
  }
};

const copy = { ...user };

copy.settings.theme = "light";

console.log(user.settings.theme);
// "light"
```

Вложенный объект остался тем же.

Лучше:

```js
const copy = {
  ...user,
  settings: {
    ...user.settings,
    theme: "light"
  }
};
```

Для глубокого клонирования иногда:

```js
const copy = structuredClone(user);
```

Но `structuredClone` не клонирует функции, DOM-ноды и некоторые специальные объекты.

---

# 33. Spread `undefined` в объекте и массиве ведёт себя по-разному

Объект:

```js
const obj = {
  ...undefined
};
// {}
```

Массив:

```js
const arr = [
  ...undefined
];
// TypeError
```

Плохо:

```js
const list = [
  ...maybeItems
];
```

Лучше:

```js
const list = [
  ...(maybeItems ?? [])
];
```

Для объектов тоже лучше явно:

```js
const props = {
  ...(maybeProps ?? {})
};
```

---

# 34. Conditional spread с `&&`

Объект обычно работает:

```js
const props = {
  id,
  ...(isDisabled && { disabled: true })
};
```

Но с массивами так нельзя:

```js
const items = [
  first,
  ...(condition && extraItems)
];
```

Если `condition === false`, будет `...false`, и код упадёт.

Лучше:

```js
const items = [
  first,
  ...(condition ? extraItems : [])
];
```

---

# 35. `in` проверяет ключ даже в prototype chain

```js
"toString" in {};
// true
```

Если нужно проверить собственное поле объекта:

```js
Object.hasOwn(obj, key)
```

Или старый вариант:

```js
Object.prototype.hasOwnProperty.call(obj, key)
```

---

# 36. Объект как map может конфликтовать с prototype keys

Плохо:

```js
const counts = {};
counts["toString"] = 1;
```

Лучше:

```js
const counts = Object.create(null);
```

Или чаще:

```js
const counts = new Map();

counts.set(key, value);
counts.get(key);
counts.has(key);
```

Если ключи не только строки — `Map` предпочтительнее.

---

# 37. `JSON.stringify` молча выкидывает часть данных

```js
JSON.stringify({
  a: undefined,
  b: () => {},
  c: Symbol("x"),
  d: 123
});
// {"d":123}
```

В массивах иначе:

```js
JSON.stringify([undefined, () => {}, Symbol("x")]);
// [null,null,null]
```

`BigInt` вообще уронит:

```js
JSON.stringify({ value: 10n });
// TypeError
```

Не стоит считать JSON универсальным клонированием.

---

# 38. `JSON.parse(JSON.stringify(obj))` как deep clone

Плохо:

```js
const copy = JSON.parse(JSON.stringify(obj));
```

Потеряются:

```js
undefined
Date как Date
Map
Set
BigInt
Infinity
NaN
RegExp
functions
symbols
class instances
```

Лучше:

```js
const copy = structuredClone(obj);
```

Но и он не для всего. Для сложных domain objects лучше явное копирование.

---

# 39. `Number.toFixed` возвращает строку

```js
const price = 10.5;

const rounded = price.toFixed(2);
// "10.50"
```

Плохо:

```js
const total = price.toFixed(2) + 5;
// "10.505"
```

Лучше:

```js
const total = Number(price.toFixed(2)) + 5;
```

Для денег лучше хранить в минимальных единицах:

```js
const priceAgorot = 1050;
```

---

# 40. Floating point арифметика

```js
0.1 + 0.2;
// 0.30000000000000004
```

Плохо:

```js
if (a + b === 0.3) {
}
```

Лучше:

```js
const EPSILON = 1e-10;

if (Math.abs(a + b - 0.3) < EPSILON) {
}
```

Для денег — integer cents/agorot.

---

# 41. `parseInt` без radix

Лучше явно:

```js
parseInt("08", 10);
```

А ещё `parseInt` обрезает хвост:

```js
parseInt("12px", 10); // 12
Number("12px");       // NaN
```

Если нужно строгое число — не использовать `parseInt` как валидатор.

---

# 42. `Math.max(...bigArray)` может упасть

Плохо:

```js
Math.max(...hugeArray);
```

Для большого массива можно получить:

```js
RangeError: Maximum call stack size exceeded
```

Лучше:

```js
let max = -Infinity;

for (const value of hugeArray) {
  if (value > max) max = value;
}
```

Или:

```js
const max = hugeArray.reduce(
  (currentMax, value) => Math.max(currentMax, value),
  -Infinity
);
```

---

# 43. `const` не делает объект immutable

```js
const user = {
  name: "Alice"
};

user.name = "Bob"; // можно
```

`const` запрещает только переназначение переменной:

```js
user = {}; // нельзя
```

Если надо запретить изменение объекта:

```js
Object.freeze(user);
```

Но это shallow freeze.

---

# 44. `switch` и забытый `break`

Плохо:

```js
switch (status) {
  case "idle":
    start();
  case "loading":
    showSpinner();
    break;
}
```

Если `status === "idle"`, выполнится и `start()`, и `showSpinner()`.

Лучше:

```js
switch (status) {
  case "idle":
    start();
    break;

  case "loading":
    showSpinner();
    break;
}
```

ESLint:

```json
{
  "rules": {
    "no-fallthrough": "error"
  }
}
```

---

# 45. Сравнение объектов по ссылке

```js
{} === {};
// false

[] === [];
// false
```

Это проверка на тот же самый объект, а не на одинаковое содержимое.

Для shallow comparison:

```js
function shallowEqual(a, b) {
  if (a === b) return true;

  const aKeys = Object.keys(a);
  const bKeys = Object.keys(b);

  if (aKeys.length !== bKeys.length) return false;

  return aKeys.every(key => Object.is(a[key], b[key]));
}
```

---

# 46. `&&` в JSX случайно рендерит `0`

Плохо:

```jsx
{items.length && <List items={items} />}
```

Если `items.length === 0`, React отрендерит `0`.

Лучше:

```jsx
{items.length > 0 && <List items={items} />}
```

Или:

```jsx
{items.length ? <List items={items} /> : null}
```

---

# 47. `&&` возвращает не boolean

```js
const result = user && user.name;
```

`result` может быть:

```js
null
undefined
false
0
""
"name"
```

Если нужен boolean:

```js
const hasName = Boolean(user?.name);
```

Но если пустая строка валидна, `Boolean("")` будет `false`.

Лучше явно:

```js
const hasName = user?.name != null;
```

---

# 48. `||` для boolean-флага ломает `false`

Плохо:

```js
const enabled = config.enabled || true;
```

Если пользователь явно передал:

```js
{ enabled: false }
```

получится `true`.

Лучше:

```js
const enabled = config.enabled ?? true;
```

Очень частая мина в настройках.

---

# 49. `setInterval` без очистки

Плохо:

```js
setInterval(() => {
  sync();
}, 1000);
```

Если это внутри компонента/инициализации, можно создать утечки.

Лучше:

```js
const intervalId = setInterval(() => {
  sync();
}, 1000);

clearInterval(intervalId);
```

В React:

```jsx
useEffect(() => {
  const intervalId = setInterval(() => {
    sync();
  }, 1000);

  return () => clearInterval(intervalId);
}, []);
```

---

# 50. `addEventListener` с anonymous function, которую потом нельзя удалить

Плохо:

```js
element.addEventListener("click", () => {
  console.log("click");
});

element.removeEventListener("click", () => {
  console.log("click");
});
```

Это две разные функции. Listener не удалится.

Лучше:

```js
function handleClick() {
  console.log("click");
}

element.addEventListener("click", handleClick);
element.removeEventListener("click", handleClick);
```

---

# 51. `event.target` vs `event.currentTarget`

Плохо:

```js
button.addEventListener("click", event => {
  event.target.disabled = true;
});
```

Если внутри кнопки есть `span`, `event.target` может быть `span`, а не кнопка.

Лучше:

```js
button.addEventListener("click", event => {
  event.currentTarget.disabled = true;
});
```

`target` — откуда событие пришло.

`currentTarget` — на чём висит обработчик.

---

# 52. `innerHTML` с пользовательскими данными

Плохо:

```js
element.innerHTML = userInput;
```

Это XSS.

Лучше:

```js
element.textContent = userInput;
```

Если нужен HTML — санитайзить через проверенную библиотеку.

# 53. `localStorage` хранит только строки

```js
localStorage.setItem("enabled", false);

const enabled = localStorage.getItem("enabled");
// "false"
```

А строка `"false"` truthy:

```js
if (enabled) {
  // выполнится
}
```

Лучше:

```js
localStorage.setItem("enabled", JSON.stringify(false));

const enabled = JSON.parse(localStorage.getItem("enabled") ?? "false");
```

С защитой:

```js
function readJson(key, fallback) {
  try {
    const raw = localStorage.getItem(key);
    return raw == null ? fallback : JSON.parse(raw);
  } catch {
    return fallback;
  }
}
```

# 54. `RegExp` с глобальным флагом `g` имеет состояние

Плохо:

```js
const regex = /foo/g;

regex.test("foo"); // true
regex.test("foo"); // false
regex.test("foo"); // true
```

У regex с `g` есть состояние `lastIndex`.

Лучше для простого `test` не использовать `g`:

```js
/foo/.test("foo");
```

Или сбрасывать:

```js
regex.lastIndex = 0;
regex.test(value);
```

# 55. `String.prototype.replace` заменяет только первое совпадение

```js
"foo foo foo".replace("foo", "bar");
// "bar foo foo"
```

Для всех:

```js
"foo foo foo".replaceAll("foo", "bar");
```

Или regex:

```js
"foo foo foo".replace(/foo/g, "bar");
```

# 56. `includes` для строки может дать ложноположительный результат

```js
"admin,user".includes("min");
// true
```

Если проверяешь роль, это баг.

Плохо:

```js
if (rolesString.includes("admin")) {
}
```

Лучше:

```js
const roles = rolesString.split(",");

if (roles.includes("admin")) {
}
```

# 57. Проверка массива через truthy

Плохо:

```js
if (items) {
  renderList(items);
}
```

Пустой массив truthy:

```js
Boolean([]); // true
```

Если нужно проверить непустоту:

```js
if (items.length > 0) {
  renderList(items);
}
```

Если массив может отсутствовать:

```js
if ((items?.length ?? 0) > 0) {
  renderList(items);
}
```

# 58. `fetch` не падает на HTTP 404/500

Плохо:

```js
const response = await fetch(url);
const data = await response.json();
```

`fetch` сам по себе не падает на HTTP 404/500. А `response.json()` может упасть, если сервер вернул не JSON или пустое тело.

Надёжнее:

```js
const response = await fetch(url);

if (!response.ok) {
  throw new Error(`Request failed: ${response.status}`);
}

const text = await response.text();
const data = text ? JSON.parse(text) : null;
```

# 59. `fetch` не имеет timeout сам по себе

Плохо:

```js
await fetch(url);
```

Может висеть долго.

Лучше:

```js
const controller = new AbortController();

const timeoutId = setTimeout(() => {
  controller.abort();
}, 10_000);

try {
  const response = await fetch(url, {
    signal: controller.signal
  });
} finally {
  clearTimeout(timeoutId);
}
```

# 60. Смешивание `||`, `&&` и `??`

Вот так нельзя без скобок:

```js
const value = a || b ?? c;
// SyntaxError
```

Нужно явно:

```js
const value = (a || b) ?? c;
```

Или:

```js
const value = a || (b ?? c);
```

Но лучше не писать слишком умные выражения:

```js
const value = user && user.settings || defaultSettings;
```

Лучше:

```js
const value = user?.settings ?? defaultSettings;
```

# 61. Нечаянная потеря `0` при форматировании

Плохо:

```js
function formatCount(count) {
  return count || "—";
}

formatCount(0); // "—"
```

Но `0` — нормальное значение.

Лучше:

```js
function formatCount(count) {
  return count ?? "—";
}
```

Если нужно заменить только пустую строку:

```js
function formatTitle(title) {
  return title === "" ? "Untitled" : title;
}
```

# 62. `catch (e)` ловит что угодно, не обязательно `Error`

```js
try {
  throw "oops";
} catch (error) {
  console.log(error.message); // undefined
}
```

Лучше:

```js
catch (error) {
  if (error instanceof Error) {
    console.error(error.message);
  } else {
    console.error("Unknown error", error);
  }
}
```

В TypeScript:

```ts
catch (error: unknown) {
  if (error instanceof Error) {
    console.error(error.message);
  }
}
```

# 63. `throw new Error({ ... })`

Плохо:

```js
throw new Error({ code: "NO_USER" });
```

Message будет:

```js
"[object Object]"
```

Лучше:

```js
throw new Error("No user");
```

С дополнительными полями:

```js
class AppError extends Error {
  constructor(message, code) {
    super(message);
    this.code = code;
  }
}

throw new AppError("No user", "NO_USER");
```

# Что реально помогает

## 1. TypeScript в строгом режиме

Минимальный полезный набор:

```json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "noImplicitOverride": true,
    "noFallthroughCasesInSwitch": true
  }
}
```

Особенно полезно:

```json
"strictNullChecks": true
```

Ловит массу багов с `null`/`undefined`.

```json
"noUncheckedIndexedAccess": true
```

Заставляет помнить, что:

```ts
arr[0]
```

может быть `undefined`.

---

## 2. ESLint

Базовый набор правил:

```json
{
  "rules": {
    "eqeqeq": ["error", "always", { "null": "ignore" }],
    "no-fallthrough": "error",
    "no-implicit-coercion": "warn",
    "no-array-constructor": "error",
    "no-new-wrappers": "error",
    "no-throw-literal": "error",
    "radix": "error",
    "require-await": "warn",
    "no-return-await": "off"
  }
}
```

Для TypeScript:

```json
{
  "rules": {
    "@typescript-eslint/no-floating-promises": "error",
    "@typescript-eslint/await-thenable": "error",
    "@typescript-eslint/no-misused-promises": "error",
    "@typescript-eslint/prefer-nullish-coalescing": "warn",
    "@typescript-eslint/prefer-optional-chain": "warn",
    "@typescript-eslint/strict-boolean-expressions": "warn",
    "@typescript-eslint/switch-exhaustiveness-check": "error"
  }
}
```

Особенно полезное, но местами раздражающее:

```json
"@typescript-eslint/strict-boolean-expressions": "warn"
```

Оно заставляет не писать мутное:

```ts
if (value) {}
```

а писать явно:

```ts
if (value != null) {}
if (value.length > 0) {}
if (value === true) {}
```

---

## 3. Нормализовать входы на границе

Не размазывать проверки по всему коду:

```js
function render(rawConfig) {
  const config = normalizeConfig(rawConfig);

  // дальше config уже нормальный
}
```

Например:

```js
function normalizeConfig(config) {
  return {
    enabled: config?.enabled ?? true,
    title: config?.title ?? "Untitled",
    items: Array.isArray(config?.items) ? config.items : []
  };
}
```

Это лучше, чем везде писать:

```js
config?.items?.map(...)
config?.title || ...
config?.enabled || ...
```

---

## 4. Явные проверки вместо “умного truthy/falsy”

Плохо:

```js
if (value) {}
```

Лучше спрашивать себя: что именно я проверяю?

```js
if (value != null) {}          // не null/undefined
if (value.length > 0) {}       // непустая строка/массив
if (value === true) {}         // именно true
if (Number.isFinite(value)) {} // нормальное число
if (Array.isArray(value)) {}   // массив
```

---

## 5. Не писать слишком плотные выражения

Плохо:

```js
const result = data?.items?.filter(Boolean).map(x => x.value || defaultValue) || [];
```

Лучше:

```js
const items = data?.items ?? [];

const result = items
  .filter(item => item != null)
  .map(item => item.value ?? defaultValue);
```

JavaScript очень любит наказывать за “элегантность”, особенно когда смешиваются `?.`, `||`, `??`, `&&`, destructuring и implicit coercion.

Главная мысль: в JS лучше не писать «коротко, потому что можно». Лучше писать так, чтобы было видно, какие значения ты считаешь допустимыми: `0`, `""`, `false`, `null`, `undefined`, `NaN`, пустой массив и пустой объект — это всё разные случаи, а язык постоянно пытается свалить их в одну кучу под названием falsy.
