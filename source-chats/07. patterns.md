# Паттерны программирования: вопросы и ответы

## Вопрос 1

Как вообще программисты пришли к идее того, что надо использовать какие-то паттерны программирования? Почему именно такие паттерны, а не другие?

## Ответ

Идея паттернов появилась не потому, что кто-то однажды решил: “давайте придумаем 23 красивых шаблона”. Скорее наоборот: программисты много лет снова и снова наступали на одни и те же грабли, а потом заметили, что некоторые решения повторяются.

То есть паттерн — это не изобретение с нуля, а **имя для уже найденного удачного решения**.

Например:

> Мы уже десять раз сталкивались с ситуацией, когда нужно создать объект, но не хотим жёстко прибивать код к конкретному классу. И каждый раз решение похоже: вынести создание объекта в отдельное место. Давайте назовём это “фабрикой”.

Или:

> У нас есть объект, за изменениями которого должны следить другие объекты. Мы уже много раз писали список подписчиков, метод `subscribe`, метод `notify`. Давайте назовём это “наблюдатель”.

Так появляются паттерны.

### Откуда сама идея “паттернов”

Идея пришла не прямо из программирования, а из архитектуры.

Архитектор Кристофер Александер писал о том, что в архитектуре городов и зданий есть повторяющиеся удачные решения: двор внутри дома, окна на солнечную сторону, переходные пространства между улицей и домом, маленькие площади и т.д.

Он называл это **patterns** — устойчивые решения устойчивых проблем.

Программисты подхватили идею: в коде ведь то же самое. Есть повторяющиеся проблемы проектирования, и есть повторяющиеся способы их решать.

Особенно известной эта идея стала после книги **Design Patterns: Elements of Reusable Object-Oriented Software**. Её написали Gamma, Helm, Johnson и Vlissides — их часто называют “Gang of Four”, или GoF.

Но важно: книга не столько “придумала” паттерны, сколько **зафиксировала и назвала практики, которые уже ходили среди программистов**.

### Почему паттерны вообще понадобились

Потому что по мере роста программ выяснилось: главная сложность не в том, как написать `if`, `for` или функцию. Главная сложность — **как организовать связи между частями программы**, чтобы она не превратилась в кашу.

Маленькая программа:

```js
const price = amount * 1.2;
console.log(price);
```

Тут паттерны почти не нужны.

Но большая программа быстро начинает задавать неприятные вопросы:

- Как создавать объекты, если их тип заранее неизвестен?
- Как менять поведение программы, не переписывая полсистемы?
- Как сделать так, чтобы один модуль не знал слишком много о другом?
- Как дать объектам общаться, но не связать их намертво?
- Как спрятать сложность за простым интерфейсом?
- Как добавлять новые варианты поведения без огромного `switch` на тысячу строк?

Вот на такие вопросы и отвечают паттерны.

### Почему именно такие паттерны, а не другие

Потому что они соответствуют типовым болезням программ.

Например, есть несколько вечных проблем.

#### 1. Создание объектов

Иногда создать объект напрямую неудобно:

```js
const transport = new DieselTruck();
```

Код теперь знает, что ему нужен именно `DieselTruck`. А если завтра будет `ElectricTruck`? Или выбор зависит от конфига?

Так появляются паттерны вроде:

```js
const transport = transportFactory.create(config);
```

Это семейство **порождающих паттернов**: Factory, Builder, Prototype, Singleton и т.д.

Они отвечают на вопрос:

> Кто и как должен создавать объекты?

#### 2. Связи между объектами

Допустим, есть кнопка, форма, состояние, валидация, сервер, уведомления. Если каждый объект напрямую знает обо всех остальных, получается клубок.

Тогда появляются паттерны вроде Observer, Mediator, Facade, Adapter.

Они отвечают на вопрос:

> Как объектам быть связанными, но не превратиться в комок проводов?

Например, Observer:

```js
store.subscribe(() => {
  render(store.getState());
});
```

Это тот же базовый мотив: “кто-то изменился — все подписчики узнали”.

#### 3. Изменяемое поведение

Иногда поведение надо подменять: разные алгоритмы сортировки, разные способы оплаты, разные политики скидок, разные способы рендера.

Плохой вариант:

```js
if (paymentType === 'card') {
  payByCard();
} else if (paymentType === 'paypal') {
  payByPayPal();
} else if (paymentType === 'crypto') {
  payByCrypto();
}
```

Более паттерновый вариант:

```js
paymentStrategy.pay(order);
```

Так появляется Strategy.

Он отвечает на вопрос:

> Как менять алгоритм, не переписывая код, который им пользуется?

### Паттерны — это кристаллизация повторяющихся решений

Паттерны можно понимать как кристаллизацию на уровне архитектуры:

```text
изменение состояния + подписчики = Observer
семейство алгоритмов + подмена = Strategy
сложное создание объекта + отдельный сборщик = Builder/Factory
несовместимые интерфейсы + прослойка = Adapter
сложная подсистема + простой вход = Facade
```

То есть паттерн появляется там, где язык сам ещё не дал готовой конструкции.

### Паттерны часто показывают недостатки языка

Иногда паттерн становится популярным не потому, что он “вечная истина”, а потому что в конкретном языке неудобно выразить какую-то идею напрямую.

Например, в классическом Java без функций как значений паттерн Strategy часто оформлялся через классы:

```java
interface SortStrategy {
    void sort(List<Item> items);
}

class QuickSortStrategy implements SortStrategy {
    public void sort(List<Item> items) {
        // ...
    }
}
```

А в JavaScript это часто просто функция:

```js
function processItems(items, sortStrategy) {
  return sortStrategy(items);
}

processItems(items, quickSort);
```

То есть то, что в одном языке выглядит как “паттерн”, в другом может быть обычной языковой конструкцией.

Ещё пример: Iterator.

В старых ООП-языках это мог быть отдельный паттерн: объект-итератор с методами `next`, `hasNext`.

А сейчас во многих языках итерация встроена:

```js
for (const item of items) {
  console.log(item);
}
```

Паттерн как бы растворился в синтаксисе языка.

### Почему не любые повторяющиеся решения становятся паттернами

Чтобы решение стало паттерном, оно должно быть не просто часто встречающимся, а ещё достаточно общим.

Например, “всегда называй переменную `data`” — это не паттерн. Это привычка, причём сомнительная.

А вот “раздели создание объекта и использование объекта” — это уже абстрактная идея, которая применима в куче разных мест.

Хороший паттерн обычно имеет:

1. **повторяющуюся проблему**;
2. **типовое решение**;
3. **понятные последствия**;
4. **цену**.

Последний пункт очень важен. Паттерн — не бесплатная магия. Он почти всегда покупает гибкость ценой сложности.

Например:

- Factory уменьшает жёсткую связь с конкретными классами, но добавляет ещё один слой.
- Observer удобно рассылает события, но может сделать поток выполнения менее очевидным.
- Strategy убирает огромные `if`, но добавляет больше сущностей.
- Singleton даёт глобальную точку доступа, но часто превращается в скрытую глобальную переменную.

### Почему паттерны особенно расцвели в ООП

Классические GoF-паттерны появились в эпоху, когда ООП было главным способом бороться со сложностью больших программ.

В ООП есть несколько центральных механизмов:

```text
объект
класс
интерфейс
наследование
композиция
инкапсуляция
полиморфизм
```

GoF-паттерны во многом являются комбинациями этих механизмов.

Например:

- Strategy — “заверни алгоритм в объект и подставляй через общий интерфейс”.
- Adapter — “оберни объект другим объектом с нужным интерфейсом”.
- Decorator — “оберни объект, добавив поведение”.
- Composite — “сделай дерево объектов, где лист и узел выглядят одинаково для пользователя”.
- Command — “заверни действие в объект”.

То есть многие классические паттерны можно грубо описать так:

> Сделай ещё один объект-посредник, чтобы отделить одну ответственность от другой.

Это очень ООП-шный способ мышления.

### А в функциональном программировании паттерны другие?

Да. В ФП многие GoF-паттерны либо исчезают, либо становятся проще, потому что функции уже являются значениями.

- Strategy? Просто передай функцию.
- Command? Часто тоже функция или описание действия как данные.
- Iterator? Ленивые последовательности, генераторы, stream/list processing.
- Visitor? Pattern matching.
- Builder? Композиция функций или immutable data transformation.

Зато появляются другие “паттерны мышления”:

```text
map/filter/reduce
pipeline
composition
currying
monad / effect wrapper
immutable update
state reducer
event sourcing
```

Например Redux — это в каком-то смысле не “GoF-паттерн”, но очень узнаваемый архитектурный паттерн:

```text
state + action + reducer = новое состояние
```

```js
function reducer(state, action) {
  switch (action.type) {
    case 'add':
      return [...state, action.item];
    default:
      return state;
  }
}
```

Это тоже паттерн, только не из классического ООП-мира.

### Коротко

Программисты пришли к паттернам потому, что в больших программах повторяются не только строки кода, но и **формы проблем**.

Паттерны — это названные формы удачных решений.

Почему именно такие? Потому что именно такие проблемы постоянно возникали в больших ООП-системах: создание объектов, управление зависимостями, подмена поведения, разделение ответственности, согласование интерфейсов, рассылка событий.

Но паттерны не вечные законы природы. Они зависят от языка, эпохи и стиля программирования. То, что в Java выглядит как паттерн из пяти классов, в JavaScript может быть одной функцией. А то, что в ООП решается через объекты, в ФП может решаться через композицию функций и неизменяемые данные.

## Вопрос 2

Ты говоришь, что некоторые паттерны растворились в конструкциях языка. Так может быть они и появились потому что в языке не было удобного способа это выразить, поэтому пришлось городить паттерн?

## Ответ

Да, именно. Это очень точное наблюдение.

Многие паттерны можно понимать так:

> **Паттерн — это “ручная конструкция языка”, которой в самом языке пока нет.**

То есть программисты многократно пишут один и тот же архитектурный трюк, потому что язык не даёт короткого, естественного способа выразить эту мысль. Потом этому трюку дают имя. А потом новый язык или новая версия языка встраивает эту идею прямо в синтаксис или стандартную библиотеку — и паттерн “растворяется”.

### Пример 1: Iterator

Когда-то “итератор” был полноценным паттерном: отдельный объект, который умеет ходить по коллекции.

Условно:

```java
Iterator<Item> it = items.iterator();

while (it.hasNext()) {
    Item item = it.next();
    process(item);
}
```

Но потом языки начали встраивать это в синтаксис:

```js
for (const item of items) {
  process(item);
}
```

Паттерн не исчез как идея. Он просто стал **языковой конструкцией**.

Теперь программисту не надо думать: “сейчас применю Iterator pattern”. Он просто пишет `for...of`.

### Пример 2: Strategy

В классическом ООП Strategy часто означает: “заверни алгоритм в объект с общим интерфейсом”.

```java
interface SortStrategy {
    void sort(List<Item> items);
}

class QuickSort implements SortStrategy {
    public void sort(List<Item> items) {
        // ...
    }
}

class MergeSort implements SortStrategy {
    public void sort(List<Item> items) {
        // ...
    }
}
```

А в JavaScript, где функции — обычные значения, это часто просто:

```js
function process(items, sortFn) {
  return sortFn(items);
}

process(items, quickSort);
```

То есть там, где Java старого стиля городила “объект-стратегию”, JavaScript говорит:

> Просто передай функцию.

Паттерн Strategy частично растворяется в **first-class functions** — функциях как значениях.

### Пример 3: Command

Command pattern — это “заверни действие в объект, чтобы выполнить его позже, положить в очередь, отменить, залогировать”.

В ООП это может выглядеть так:

```java
interface Command {
    void execute();
}

class SaveFileCommand implements Command {
    public void execute() {
        saveFile();
    }
}
```

А в языке с функциями как значениями простейший command — это просто функция:

```js
const command = () => saveFile();

queue.push(command);

// позже
queue.shift()();
```

Опять же: паттерн не полностью исчез, особенно если нужны `undo`, сериализация, история команд. Но его простая форма стала настолько естественной, что мы уже не называем её паттерном.

### Пример 4: Observer

Observer — “объект изменился, подписчики получили уведомление”.

Раньше это писали руками:

```js
class Store {
  listeners = [];

  subscribe(fn) {
    this.listeners.push(fn);
  }

  notify() {
    for (const fn of this.listeners) {
      fn();
    }
  }
}
```

А потом появились event emitter, DOM events, reactive streams, signals, observables, hooks, state libraries.

Например:

```js
button.addEventListener('click', handleClick);
```

Это по сути Observer, но он встроен в платформу.

Ты не говоришь:

> Сейчас я применю паттерн Observer.

Ты говоришь:

> Повешу обработчик события.

Паттерн стал частью среды.

### Пример 5: Visitor и pattern matching

Visitor pattern часто нужен, когда у тебя есть набор разных типов объектов, и ты хочешь выполнять разные действия в зависимости от конкретного типа.

В ООП это может быть громоздко:

```java
interface Node {
    void accept(Visitor visitor);
}

class NumberNode implements Node {
    void accept(Visitor visitor) {
        visitor.visitNumber(this);
    }
}

class AddNode implements Node {
    void accept(Visitor visitor) {
        visitor.visitAdd(this);
    }
}
```

А в языках с нормальным `match` это выражается гораздо прямее:

```rust
match node {
    Node::Number(value) => ...
    Node::Add(left, right) => ...
}
```

И снова: то, что в одном языке является паттерном с кучей классов, в другом является обычной управляющей конструкцией.

### Можно сформулировать жёстче

Многие классические ООП-паттерны — это **компенсация ограничений ООП-языков своего времени**.

Особенно ограничений вроде:

```text
нет функций как обычных значений
нет алгебраических типов данных
нет pattern matching
слабая работа с композициями
много ceremony вокруг классов и интерфейсов
нет удобных модулей
нет удобных генераторов/итераторов
```

Поэтому программисты строили недостающие конструкции из того, что было под рукой:

```text
классы
интерфейсы
наследование
композиция
виртуальные методы
```

Отсюда ощущение, что многие GoF-паттерны — это “обходные дороги” вокруг дырок в языке.

### Но есть важная оговорка

Не все паттерны — это просто костыли языка.

Некоторые паттерны выражают более глубокие архитектурные идеи, которые не исчезают даже в хороших языках.

#### Facade

Сложную подсистему часто полезно закрыть простым интерфейсом:

```js
const report = await analytics.generateMonthlyReport(userId);
```

Внутри может быть база, кэш, API, очередь, форматирование, права доступа. Язык не отменяет необходимости прятать сложность.

#### Adapter

Если у двух систем разные интерфейсы, всё равно нужен переходник:

```js
const normalizedUser = adaptExternalUser(apiUser);
```

Даже самый прекрасный язык не сделает чужой API автоматически таким, как тебе нужно.

#### Boundary / Layer / Ports and Adapters

Разделение “доменной логики” и “внешнего мира” — база архитектуры. Это не просто недостаток языка. Это свойство больших систем: база данных, сеть, UI, файлы, время, случайность — всё это надо как-то отделять от чистой логики.

### Поэтому лучше так

Не “паттерны появились только из-за плохих языков”.

А:

> **Часть паттернов — это симптомы отсутствующих языковых конструкций. Часть паттернов — это устойчивые архитектурные формы, которые нужны даже в хороших языках.**

Например:

```text
Iterator → часто растворился в for/of, generators, Iterable
Strategy → часто растворился в функциях как значениях
Command → частично растворился в функциях/closures
Visitor → часто растворяется в pattern matching
Observer → частично растворился в events/signals/reactivity
Factory → частично растворяется в DI/container/config/modules, но не исчезает
Facade → почти не исчезает
Adapter → почти не исчезает
State machine → не исчезает, только получает лучшие формы записи
```

### Хорошая проверка

Можно спросить:

> Этот паттерн нужен потому, что в предметной области реально есть такая форма, или потому, что язык заставляет меня изображать её вручную?

Например:

```js
items.map(x => x * 2)
```

Тут не нужен паттерн “обход коллекции и накопление результата”. Язык уже дал `map`.

А вот:

```js
function adaptOldTelemetryPacket(packet) {
  return {
    x: packet.coord_x,
    y: packet.coord_y,
    timestamp: Number(packet.ts),
  };
}
```

Это Adapter, и он нужен не потому, что JavaScript плохой. Он нужен потому, что внешний мир прислал данные не в том виде, в каком живёт твоя программа.

### Коротко

Хороший язык впитывает удачные паттерны. Но мир всё равно сложнее любого языка, поэтому новые паттерны появляются снова.

Паттерны возникают на границе между выразительной силой языка и сложностью реального мира.

## Вопрос 3

Мне понравилось про то, что некоторые паттерны решают какую-то часто повторяющуюся боль программистов. А пожалуйста напиши список типичных юзкейсов, а к каждому паттерн.

## Ответ

Удобнее думать не от названия паттерна, а от боли:

> “У меня в коде снова появилась вот такая проблема. Какую форму решения обычно применяют?”

Ниже список именно в таком виде.

## Типичные боли → подходящие паттерны

| Боль / юзкейс | Паттерн | Смысл |
|---|---|---|
| Нужно создавать объекты, но код не должен знать конкретный класс | **Factory Method / Abstract Factory** | Вынести создание объектов в отдельное место |
| Создание объекта сложное: много параметров, опциональные шаги, разные конфигурации | **Builder** | Собирать объект по шагам |
| Нужен ровно один общий экземпляр чего-то: конфиг, логгер, registry | **Singleton** | Один глобально доступный объект, но применять осторожно |
| Нужно скопировать существующий объект как шаблон | **Prototype** | Создавать новые объекты копированием образца |
| Есть несовместимый интерфейс, а хочется использовать его как свой | **Adapter** | Сделать “переходник” |
| Есть сложная подсистема, а снаружи хочется один простой вход | **Facade** | Закрыть сложность простым API |
| Нужно добавить поведение объекту, не меняя его класс | **Decorator** | Обернуть объект другим объектом |
| Есть дерево объектов, но хочется работать с листом и группой одинаково | **Composite** | Сделать единый интерфейс для одиночных и составных объектов |
| Объект дорогой или удалённый, нужен представитель вместо него | **Proxy** | Подставить посредника, который контролирует доступ |
| Много похожих мелких объектов, хочется экономить память | **Flyweight** | Разделять общее состояние между объектами |
| Нужно подменять алгоритм: разные сортировки, способы оплаты, правила скидок | **Strategy** | Передавать поведение как заменяемую стратегию |
| Нужно выполнить действие позже, положить его в очередь, повторить или отменить | **Command** | Завернуть действие в объект/функцию |
| Один объект изменился, другие должны узнать | **Observer** | Подписчики получают уведомления |
| Есть сложный процесс с фиксированным скелетом, но отдельные шаги меняются | **Template Method** | Общий алгоритм в базовом классе, детали в наследниках |
| Нужно пройти по коллекции, не раскрывая её внутреннее устройство | **Iterator** | Отдельный механизм обхода |
| Нужно менять поведение объекта в зависимости от состояния | **State** | Состояния становятся отдельными объектами/ветками логики |
| Есть цепочка обработчиков: один не справился — передаёт дальше | **Chain of Responsibility** | Последовательность обработчиков |
| Много объектов общаются друг с другом, связи превращаются в паутину | **Mediator** | Центральный посредник управляет коммуникацией |
| Нужно сохранить состояние объекта, чтобы потом откатиться | **Memento** | Снимок состояния |
| Нужно добавить операции к разным типам объектов, не меняя сами классы | **Visitor** | Вынести операцию наружу |
| Нужно интерпретировать маленький язык, правила, формулы, выражения | **Interpreter** | Представить грамматику в виде объектов/дерева |
| Разные части программы должны получать зависимость, но не создавать её сами | **Dependency Injection** | Передавать зависимости снаружи |
| Нужно отделить бизнес-логику от базы, UI, сети и файлов | **Ports and Adapters / Hexagonal Architecture** | Внутри чистая логика, снаружи адаптеры |
| Нужно централизованно менять состояние приложения | **Reducer / Flux / Redux-style** | `state + action => newState` |
| Нужно описать долгоживущий процесс: заказ, доставка, регистрация, платежи | **State Machine / Saga** | Явно моделировать шаги и переходы |
| Нужно реагировать на поток событий во времени | **Event-driven / Pub-Sub / Reactive** | События как основа взаимодействия |
| Нужно не вызывать тяжёлую операцию слишком часто | **Debounce / Throttle** | Ограничить частоту вызовов |
| Нужно не пересчитывать одно и то же | **Memoization / Cache** | Запоминать результат |
| Нужно последовательно преобразовывать данные шаг за шагом | **Pipeline** | Цепочка трансформаций |
| Нужно отделить чтение данных от изменения данных | **CQRS** | Отдельные модели для команд и запросов |
| Нужно хранить не только текущее состояние, но историю изменений | **Event Sourcing** | Сохранять события, а состояние восстанавливать из них |

### Несколько самых частых на практике

#### “У меня огромный `if/else` или `switch`, который выбирает поведение”

Пример боли:

```ts
if (paymentType === 'card') {
  payByCard(order);
} else if (paymentType === 'paypal') {
  payByPayPal(order);
} else if (paymentType === 'bank') {
  payByBankTransfer(order);
}
```

Обычно сюда просится **Strategy**.

```ts
const paymentStrategies = {
  card: payByCard,
  paypal: payByPayPal,
  bank: payByBankTransfer,
};

paymentStrategies[paymentType](order);
```

Смысл: разные алгоритмы становятся взаимозаменяемыми.

#### “Код создания объекта расползся по всей программе”

Боль:

```ts
const client = new ApiClient(
  config.baseUrl,
  config.token,
  new RetryPolicy(3),
  new Logger(),
);
```

И так в десяти местах. Потом меняется логика создания — приходится чинить десять мест.

Сюда просится **Factory**.

```ts
function createApiClient(config: Config) {
  return new ApiClient({
    baseUrl: config.baseUrl,
    token: config.token,
    retryPolicy: new RetryPolicy(3),
    logger: new Logger(),
  });
}

const client = createApiClient(config);
```

Смысл: пользователь объекта не обязан знать, как именно объект собирается.

#### “У внешней библиотеки неудобный интерфейс”

Например, API отдаёт так:

```ts
const apiUser = {
  user_id: '42',
  full_name: 'Ada Lovelace',
  created_at: '2026-05-07T10:00:00Z',
};
```

А в программе принято так:

```ts
type User = {
  id: string;
  name: string;
  createdAt: Date;
};
```

Сюда просится **Adapter**.

```ts
function adaptUser(apiUser: ApiUser): User {
  return {
    id: apiUser.user_id,
    name: apiUser.full_name,
    createdAt: new Date(apiUser.created_at),
  };
}
```

Смысл: внешний мир не должен заражать внутреннюю модель программы.

#### “Слишком сложная подсистема, хочется один нормальный метод”

Например, чтобы отправить отчёт, надо:

```ts
const raw = await db.loadData();
const normalized = normalize(raw);
const pdf = renderPdf(normalized);
const email = createEmail(pdf);
await smtp.send(email);
```

Если это повторяется, напрашивается **Facade**.

```ts
await reportService.sendMonthlyReport(userId);
```

Смысл: снаружи простой вход, внутри сложная кухня.

#### “Один объект изменился, остальные должны узнать”

Например, состояние поменялось — UI должен перерендериться, логгер записать событие, аналитика отправить метрику.

Сюда просится **Observer / Pub-Sub**.

```ts
store.subscribe(render);
store.subscribe(logStateChange);
store.subscribe(sendAnalytics);

store.setState(newState);
```

Смысл: источник события не обязан знать всех, кто на него реагирует.

#### “Нужно сделать действие сейчас, позже, повторить или отменить”

Например, редактор изображений: нарисовать линию, удалить объект, переместить слой. Эти действия хочется складывать в историю.

Сюда просится **Command**.

```ts
type Command = {
  execute(): void;
  undo(): void;
};

history.push(command);
command.execute();
```

Смысл: действие становится объектом. Его можно хранить, отменять, повторять, логировать.

#### “Объект ведёт себя по-разному в разных состояниях”

Например, заказ:

```text
draft → paid → shipped → delivered
              ↘ cancelled
```

Плохой вариант — огромные условия:

```ts
if (order.status === 'draft') {
  // ...
} else if (order.status === 'paid') {
  // ...
} else if (order.status === 'shipped') {
  // ...
}
```

Сюда просится **State** или **State Machine**.

```ts
const transitions = {
  draft: ['paid', 'cancelled'],
  paid: ['shipped', 'cancelled'],
  shipped: ['delivered'],
  delivered: [],
  cancelled: [],
};
```

Смысл: состояния и разрешённые переходы становятся явной моделью, а не размазанными `if` по коду.

#### “Есть цепочка проверок или обработчиков”

Например, запрос проходит через:

```text
auth → rate limit → validation → business logic → response
```

Сюда просится **Chain of Responsibility**.

```ts
const pipeline = [
  checkAuth,
  checkRateLimit,
  validateInput,
  handleRequest,
];

for (const step of pipeline) {
  await step(context);
}
```

Смысл: каждый обработчик делает свою часть и может передать управление дальше.

#### “Мне надо добавить поведение, но не хочется менять исходный объект”

Например, есть функция запроса:

```ts
fetchData();
```

Хочется добавить логирование, кеш, retry, метрики.

Сюда просится **Decorator**.

```ts
const fetchWithLogging = withLogging(fetchData);
const fetchWithRetry = withRetry(fetchWithLogging);
const fetchCached = withCache(fetchWithRetry);
```

Смысл: поведение наращивается слоями.

#### “Есть много модулей, и они все знают друг о друге”

Боль:

```text
A вызывает B
B вызывает C
C вызывает A
A знает про D
D знает про B
```

Получается клубок.

Сюда просится **Mediator**.

```ts
mediator.send('user:created', user);
```

Смысл: объекты общаются не напрямую, а через посредника.

#### “Нужно отделить чистую логику от грязного внешнего мира”

Например, бизнес-логика не должна зависеть от того, где лежат данные: в PostgreSQL, localStorage, REST API или файле.

Сюда просится **Ports and Adapters**.

```ts
type UserRepository = {
  findById(id: string): Promise<User>;
  save(user: User): Promise<void>;
};

function createUserService(repo: UserRepository) {
  return {
    async renameUser(id: string, name: string) {
      const user = await repo.findById(id);
      return repo.save({ ...user, name });
    },
  };
}
```

Смысл: логика зависит от интерфейса, а не от конкретной базы или API.

Паттерны обычно возникают вокруг нескольких вечных вопросов.


#### Кто с кем говорит?

```text

```

#### Как менять поведение?

```text

```


### Самая практичная эвристика

Паттерн нужен не тогда, когда ты знаешь его название, а когда код начинает подавать характерный запах.

Например:

| Запах в коде | Часто помогает |
|---|---|
| Один огромный `switch` по типу действия | Strategy / Command / State |
| Везде одинаковая сложная сборка объекта | Factory / Builder |
| Модуль знает слишком много деталей другого модуля | Facade / Adapter |
| Внешний API протёк внутрь приложения | Adapter / Anti-Corruption Layer |
| Много ручных вызовов “после изменения обнови ещё вот это” | Observer |
| Поведение добавляется флагами `enableX`, `withY`, `useZ` | Decorator / Strategy |
| Состояния размазаны по boolean-флагам | State Machine |
| Трудно тестировать из-за базы, сети, времени | Dependency Injection / Ports and Adapters |
| Изменения состояния непредсказуемы | Reducer / Event Sourcing / Immutable updates |

Главное: паттерн — это не украшение архитектуры, а лекарство от конкретной повторяющейся боли. Если боли нет, паттерн часто только утяжеляет код.

## Вопрос 4

Паттерны действительно уменьшают сложность или просто прячут её в другое место?

## Ответ

Это важный вопрос, потому что паттерн часто выглядит как аккуратное решение, но иногда он не уничтожает сложность, а **перекладывает её туда, где она меньше мешает**.

### Паттерн как контейнер для сложности

Например, был код:

```js
if (type === 'card') payByCard();
if (type === 'paypal') payByPayPal();
if (type === 'bank') payByBank();
```

Мы применили Strategy:

```js
strategies[type].pay();
```

Стало короче и чище. Но сложность не исчезла. Она переехала в таблицу стратегий, регистрацию стратегий, интерфейсы, тесты, жизненный цикл объектов.

Значит, хороший паттерн не обязательно уничтожает сложность. Он должен **поместить её туда, где она меньше всего мешает**.

Например:

```text
условия выбора алгоритма → в Strategy
грязный внешний API → в Adapter
сложная подсистема → за Facade
переходы состояний → в State Machine
создание объектов → в Factory
```

То есть паттерн — это не столько “упрощатель”, сколько **контейнер для сложности**.

## Вопрос

Почему один и тот же приём сначала считается паттерном, а потом начинает выглядеть как антипаттерн?

## Ответ

Хороший пример — Singleton.

Сначала идея кажется разумной:

> У нас должен быть один логгер, конфиг или connection pool. Давайте сделаем Singleton.

Но потом Singleton часто превращается в скрытую глобальную переменную:

```js
const config = Config.getInstance();
```

Функция вроде бы ничего не принимает:

```js
function calculatePrice(order) {
  const tax = Config.getInstance().tax;
  return order.price * tax;
}
```

Но на самом деле она зависит от глобального состояния. Её труднее тестировать, труднее запускать параллельно, труднее понять.

Получается странная вещь:

```text
паттерн → когда решает реальную проблему
антипаттерн → когда применяется по инерции
```

Паттерн портится, когда его начинают использовать не как ответ на боль, а как знак “правильной архитектуры”.

### Почему паттерны исчезают в маленьком коде и возвращаются в большом

В маленькой программе Factory может выглядеть смешно:

```js
function createUser(name) {
  return { name };
}
```

Зачем ради этого отдельная функция?

Но в большой системе создание пользователя может включать:

```text
валидацию
нормализацию
дефолтные настройки
роль
права
привязку к организации
событие user.created
логирование
метрики
```

И внезапно Factory уже не кажется лишней.

То есть паттерны зависят не только от языка, но и от **масштаба программы**.

На малом масштабе паттерн выглядит как бюрократия. На большом масштабе отсутствие паттерна выглядит как хаос.

## Вопрос

Можно ли сделать так, чтобы архитектурная идея охранялась не только дисциплиной программиста, но и системой типов?

## Ответ

Например, State Machine можно держать “в голове”:

```js
if (status === 'paid') {
  shipOrder();
}
```

А можно выразить состояния типами так, чтобы неправильный переход было трудно написать:

```ts
type DraftOrder = { status: 'draft' };
type PaidOrder = { status: 'paid'; paidAt: Date };
type ShippedOrder = { status: 'shipped'; shippedAt: Date };

function ship(order: PaidOrder): ShippedOrder {
  return {
    status: 'shipped',
    shippedAt: new Date(),
  };
}
```

Теперь нельзя случайно отправить `DraftOrder`, если TypeScript достаточно строго настроен.

Это уже не просто “паттерн State”. Это попытка сделать так, чтобы сама система типов охраняла архитектурную идею.

Эволюция может выглядеть так:

```text
хаос в коде
↓
повторяющийся приём
↓
паттерн с названием
↓
библиотека
↓
языковая конструкция
↓
проверка на уровне типов/компилятора
```

Удачные формы мышления постепенно твердеют: сначала в соглашения, потом в библиотеки, потом в синтаксис, потом в типы.
