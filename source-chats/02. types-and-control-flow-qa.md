# Типы данных и управляющие конструкции — Q/A

## Q1. Откуда вообще решили, что в языках программирования должны быть какие-то определённые типы данных?

**Вопрос:**

> А откуда вообще решили, что в языках программирования должны быть какие-то определенные типы данных? Ведь для самого процессора это просто биты и всё. А почему решили выбрать именно эти управляющие конструкции (if/else/while и тд)? Ведь для процессора это просто разные варианты GOTO

**Ответ:**

Да, для процессора «всё просто биты». Но для программиста и компилятора это не просто биты.

Типы и управляющие конструкции появились как способ сказать:

```text
вот эти биты надо понимать как число
вот эти — как адрес
вот эти — как символ
а вот здесь не произвольный прыжок, а повторение до условия
```

То есть языки программирования не столько копируют процессор, сколько строят над ним человеческую модель мышления.

### Почему появились типы данных

Процессор видит, условно:

```text
01000001
```

Но что это?

```text
65          // число
'A'         // символ A
адрес       // указатель на память
часть float // кусок числа с плавающей точкой
инструкция // машинная команда
цвет        // например, яркость красного канала
```

Биты сами не говорят, как их понимать. Тип данных — это договорённость о смысле битов и допустимых операциях над ними.

Например:

```js
10 + 20
```

и:

```js
"10" + "20"
```

Внутри это тоже биты, но смысл операции разный:

```text
числа:    10 + 20  => 30
строки:  "10"+"20" => "1020"
```

Тип говорит компилятору или интерпретатору:

```text
эти данные надо складывать как числа
```

или:

```text
эти данные надо склеивать как текст
```

Исторически это быстро стало нужно, потому что первые высокоуровневые языки писались не ради красоты, а ради инженерной практики: считать траектории, зарплаты, таблицы, научные вычисления, тексты.

Например:

```c
int age = 12;
char letter = 'A';
float temperature = 36.6;
```

Все три значения в памяти — биты. Но операции над ними разные:

```c
age + 1              // осмысленно
temperature / 2      // осмысленно
letter / temperature // подозрительно
```

Типы — это ещё и защита от бессмыслицы.

### Тип — это не только форма данных

Тип — это не только то, как данные лежат в памяти. Тип ещё говорит, что с ними можно делать.

Например:

```ts
type User = {
  name: string;
  age: number;
};
```

С этим значением можно делать:

```ts
user.name.toUpperCase();
user.age + 1;
```

А вот так нельзя или странно:

```ts
user.age.toUpperCase();
user.name - 10;
```

Типы — это попытка заранее поймать ошибку:

```text
ты обращаешься с этой штукой не так, как договорился
```

### Почему именно такие типы: number, string, boolean, array

Потому что они соответствуют самым частым человеческим понятиям при решении задач.

#### Числа

Нужны почти сразу:

```js
price * count;
x + velocity * time;
```

Но чисел несколько видов:

```text
integer — целые числа
float/real — приблизительные вещественные числа
```

Целое число и число с плавающей точкой имеют разное внутреннее представление.

Отсюда классическая боль:

```js
0.1 + 0.2;
// 0.30000000000000004
```

#### Boolean

`true` / `false` появились потому, что программы постоянно задают вопросы:

```js
if (temperature > 100) {
  stop();
}
```

Выражение:

```js
temperature > 100
```

возвращает ответ:

```text
да / нет
истина / ложь
true / false
```

Можно было бы использовать `0` и `1`, но человеку удобнее мыслить так:

```js
isDoorOpen;
hasEnergy;
isEnemyVisible;
```

#### String

Текст нужен для общения с человеком:

```js
"Ошибка: файл не найден";
"Введите имя";
```

Физически строка — массив кодов символов. Но концептуально это отдельный тип, потому что операции другие:

```js
"hello".toUpperCase();
"hello".includes("ell");
```

#### Array / list

Массивы появились потому, что понадобилось хранить много однотипных значений:

```js
temperatures = [20, 21, 19, 23];
```

Без массива пришлось бы писать:

```js
temperature1;
temperature2;
temperature3;
temperature4;
```

А это тупик.

### Почему появились if/else

Почти любая программа должна выбирать поведение по ситуации:

```js
if (battery < 10) {
  returnToBase();
} else {
  continueMission();
}
```

На уровне машины это `GOTO`:

```text
если условие ложно — прыгнуть туда
если истинно — продолжить здесь
```

Но `if` даёт человеку понятную форму:

```text
если это правда — сделай это
иначе — сделай то
```

### Почему появились while/for

Циклы — это тоже облагороженный `GOTO`.

Например:

```js
while (energy > 0) {
  move();
  energy -= 1;
}
```

На машинном уровне это примерно:

```asm
LOOP:
  compare energy, 0
  jump_if_less_or_equal END
  call move
  subtract energy, 1
  jump LOOP
END:
```

То есть внутри есть прыжок назад. Но `while` говорит сразу:

```text
повторяй, пока условие истинно
```

А не:

```text
прыгни на метку LOOP, потом проверь регистр, потом прыгни на END, потом опять прыгни назад
```

`for` появился как удобная форма для частого случая: пройти по диапазону или коллекции.

```js
for (let i = 0; i < items.length; i++) {
  process(items[i]);
}
```

### Почему не оставить один GOTO

Потому что `GOTO` слишком мощный. Он позволяет прыгнуть куда угодно.

В маленьком примере это терпимо. В большой программе появляется «спагетти-код»:

```text
сюда прыгаем отсюда
туда прыгаем оттуда
а сюда иногда попадаем из середины другого процесса
```

С `if/while/for` ты видишь блок:

```js
while (condition) {
  // ...
}
```

И примерно понимаешь:

```text
вход здесь
выход здесь
повторение здесь
```

А с произвольными `GOTO` у блока может быть много тайных входов и выходов.

### Управляющие конструкции — это безопасные шаблоны GOTO

```text
if/else  = безопасный шаблон условного перехода
while    = безопасный шаблон прыжка назад
for      = безопасный шаблон счётного повторения
function = безопасный шаблон прыжка с возвратом
switch   = безопасный шаблон выбора из многих веток
```

Языки не отменили `GOTO`. Они его одомашнили.

```text
дикий GOTO: прыгни куда угодно
структурные конструкции: прыгни только по понятной схеме
```

### Почему именно if/else/while

Потому что эти конструкции соответствуют базовым формам алгоритмов:

```text
последовательность
выбор
повторение
```

Например:

```js
doA();

if (condition) {
  doB();
} else {
  doC();
}

while (!finished) {
  doD();
}
```

Можно было бы писать всё через `GOTO`, но это как писать роман только инструкциями:

```text
перейди к строке 182
если герой жив, перейди к строке 41
иначе перейди к строке 203
```

Работает. Но читать невозможно.

### Короткая формулировка

Типы и управляющие конструкции — это компрессия смысла.

Без типов:

```text
вот 32 бита
сделай с ними что-нибудь
```

С типами:

```text
это целое число
это строка
это список пользователей
это функция, принимающая число и возвращающая boolean
```

Без управляющих конструкций:

```text
прыгни туда
прыгни сюда
проверь регистр
прыгни назад
```

С управляющими конструкциями:

```text
если
пока
для каждого
вызови функцию
верни результат
```

Язык программирования — это не просто способ заставить процессор работать. Это способ сжать машинный хаос до человеческих понятий.

Для книги можно сформулировать так:

> Процессору всё равно, мальчик. Для него число, буква, команда и адрес — просто узоры из нулей и единиц. Но человеку не всё равно. Человек не может строить цивилизацию из голых битов. Ему нужны имена, типы, условия, повторения. Тип данных — это обещание о смысле битов. `if` — это приручённый прыжок. `while` — это прыжок, которому надели ошейник. А язык программирования — это договор между человеческой мыслью и машинной слепотой.

## Q2. Какие вообще типы данных бывают? И почему алгоритм описывают через последовательность, выбор и повторение? Разве нет функций?

**Вопрос:**

> Какие вообще типы данных бывают? Я знаю, что бывают примитивные и сложные. А ещё?
>
> Вот ты пишешь:
> "Почти любой алгоритм можно описать через три вещи:
> последовательность
> выбор
> повторение"
> Почему так? Разве нет других элементов? Функции?

**Ответ:**

«Примитивные и сложные» — это только одна из классификаций. Типы можно делить по разным признакам.

`number`, `string`, `Array`, `User`, `Promise`, `Option`, `Function` — всё это типы, но они живут на разных этажах.

### 1. Примитивные и составные типы

**Примитивные** — значения, которые обычно считаются атомарными:

```ts
number;
string;
boolean;
null;
undefined;
symbol;
bigint;
```

Примитивные типы отвечают на вопрос:

```text
какое это базовое значение?
```

Например:

```ts
const age: number = 12;
const name: string = "Tom";
const alive: boolean = true;
```

**Составные**, или сложные, типы строятся из других типов:

```ts
number[];
{ name: string; age: number };
Map<string, User>;
Promise<Result>;
```

Например:

```ts
type User = {
  name: string;
  age: number;
};
```

`User` — это не новый машинный атом, а комбинация других типов.

### 2. Скалярные и структурные типы

**Скалярный тип** — одно значение:

```ts
number;
boolean;
string;
```

**Структурный тип** — значение с внутренним устройством:

```ts
type Point = {
  x: number;
  y: number;
};

type Line = {
  from: Point;
  to: Point;
};
```

На уровне машины это память. На уровне программы — точка или линия.

### 3. Коллекции

Типы-контейнеры:

```ts
number[];
Array<string>;
Set<number>;
Map<string, User>;
```

Они отвечают не только на вопрос «что это?», но и:

```text
сколько там элементов?
есть ли порядок?
можно ли быстро искать?
могут ли быть дубликаты?
```

Например:

```ts
const list: string[] = ["a", "b", "c"];
const set: Set<string> = new Set(["a", "b", "c"]);
```

Оба хранят строки, но смысл разный:

```text
Array — последовательность
Set   — множество уникальных значений
```

### 4. Записи / объекты / структуры

Это типы, где значение состоит из именованных полей.

```ts
type Book = {
  title: string;
  pages: number;
  author: string;
};
```

Такие типы нужны, когда мы хотим собрать несколько связанных данных в одну сущность.

### 5. Перечисления: enum / union of literals

Иногда тип нужен не для хранения сложной структуры, а для ограничения возможных вариантов.

```ts
type Direction = "left" | "right" | "up" | "down";
```

Теперь нельзя случайно передать:

```ts
move("banana");
```

Потому что `"banana"` не является направлением.

Такие типы отвечают на вопрос:

```text
какие варианты вообще допустимы?
```

### 6. Union-типы: «или то, или это»

В TypeScript это очень важная штука.

```ts
type Id = string | number;
```

Значение типа `Id` может быть либо строкой, либо числом.

Более интересный пример:

```ts
type LoadingState =
  | { status: "loading" }
  | { status: "success"; data: User[] }
  | { status: "error"; error: Error };
```

Это говорит:

```text
если status = "loading" — данных ещё нет
если status = "success" — есть data
если status = "error" — есть error
```

И TypeScript может понимать это:

```ts
function render(state: LoadingState) {
  if (state.status === "success") {
    return state.data.map(user => user.name);
  }

  if (state.status === "error") {
    return state.error.message;
  }

  return "Loading...";
}
```

Это называется **discriminated union** — размеченное объединение.

Очень полезная альтернатива куче nullable-полей:

```ts
type BadState = {
  loading: boolean;
  data?: User[];
  error?: Error;
};
```

В таком плохом варианте возможны бессмысленные состояния:

```ts
{
  loading: true,
  data: [...],
  error: new Error()
}
```

А в union-типаже можно запретить невозможные комбинации.

### 7. Intersection-типы: «и то, и это»

```ts
type WithId = {
  id: string;
};

type WithName = {
  name: string;
};

type User = WithId & WithName;
```

`User` должен иметь и `id`, и `name`.

```ts
const user: User = {
  id: "123",
  name: "Alice",
};
```

Если union — это **или**, то intersection — это **и**.

```ts
A | B // A или B
A & B // A и B одновременно
```

### 8. Функциональные типы

Функция — это тоже значение. А значит, у неё тоже есть тип.

```ts
type Predicate = (x: number) => boolean;
```

Это тип функции, которая принимает число и возвращает `boolean`.

Например:

```ts
const isEven: Predicate = x => x % 2 === 0;
```

Функциональный тип отвечает на вопрос:

```text
какие аргументы принимает функция и что возвращает?
```

Функцию можно передать, вернуть, сохранить в переменную.

```ts
function filterUsers(
  users: User[],
  predicate: (user: User) => boolean
) {
  return users.filter(predicate);
}
```

### 9. Nullable / optional-типы

Во многих языках есть идея: значение может быть, а может отсутствовать.

В JavaScript/TypeScript это часто:

```ts
string | null;
string | undefined;
```

Например:

```ts
type User = {
  name: string;
  email?: string;
};
```

`email?: string` означает:

```text
email может быть строкой
а может отсутствовать
```

В более строгих языках это часто оформляют как специальный тип:

```rust
Option<String>
```

или:

```haskell
Maybe String
```

Смысл:

```text
либо есть значение
либо его нет
```

### 10. Result / Either-типы

Есть типы для результата операции, которая может завершиться успешно или ошибкой.

Условно:

```ts
type Result<T, E> =
  | { ok: true; value: T }
  | { ok: false; error: E };
```

Например:

```ts
function parseNumber(input: string): Result<number, string> {
  const n = Number(input);

  if (Number.isNaN(n)) {
    return { ok: false, error: "Not a number" };
  }

  return { ok: true, value: n };
}
```

Теперь ошибка — не скрытый выброс исключения, а часть типа.

```ts
const result = parseNumber("123");

if (result.ok) {
  console.log(result.value);
} else {
  console.log(result.error);
}
```

### 11. Generic-типы

Generic — это тип с параметром.

```ts
Array<string>;
Array<number>;
Promise<User>;
Map<string, number>;
```

`Array<T>` означает:

```text
массив чего-то
```

А что именно внутри — задаётся параметром.

Можно написать свой generic:

```ts
type Box<T> = {
  value: T;
};
```

Теперь:

```ts
const numberBox: Box<number> = { value: 123 };
const stringBox: Box<string> = { value: "hello" };
```

Generic-типы нужны, когда структура одна и та же, но содержимое разное.

```ts
type ApiResponse<T> = {
  data: T;
  status: number;
};
```

### 12. Номинальные и структурные типы

Это уже не про форму типа, а про то, как язык решает совместимость типов.

В TypeScript типизация в основном структурная.

```ts
type Point = {
  x: number;
  y: number;
};

type Vector = {
  x: number;
  y: number;
};
```

Для TypeScript `Point` и `Vector` совместимы, потому что структура одинаковая:

```ts
const point: Point = { x: 1, y: 2 };
const vector: Vector = point; // нормально
```

А в номинальной типизации важны имена типов.

```text
Point и Vector разные,
даже если внутри оба имеют x и y
```

Структурная типизация говорит:

```text
если выглядит как утка и крякает как утка — это утка
```

Номинальная типизация говорит:

```text
утка — только если она объявлена как Duck
```

### 13. Статические и динамические типы

**Статическая типизация** — типы проверяются до запуска программы.

Например TypeScript, Rust, Java, C#:

```ts
const x: number = "hello";
// ошибка до запуска
```

**Динамическая типизация** — типы проверяются во время выполнения.

Например JavaScript, Python, Ruby:

```js
let x = "hello";
x = 123;
```

В динамических языках типы всё равно есть. Просто они принадлежат в первую очередь значениям, а не переменным.

```js
typeof 123;      // "number"
typeof "hello"; // "string"
typeof true;     // "boolean"
```

### 14. Сильная и слабая типизация

Это мутное деление, но полезное интуитивно.

**Слабая типизация** чаще автоматически приводит значения:

```js
"5" - 1; // 4
"5" + 1; // "51"
```

JavaScript здесь особенно коварен.

**Более сильная типизация** чаще говорит:

```text
нет, строку и число нельзя просто так смешивать
скажи явно, что ты хочешь
```

Например в Python:

```python
"5" + 1
# TypeError
```

Нужно явное преобразование:

```python
int("5") + 1
```

### 15. Типы-единицы измерения

В некоторых языках или библиотеках можно моделировать не просто `number`, а «число в метрах», «число в секундах», «число в килограммах».

Условно:

```ts
type Meters = number;
type Seconds = number;
```

Но в TypeScript это само по себе слабовато, потому что оба всё равно `number`. Поэтому делают branded types:

```ts
type Brand<T, Name extends string> = T & { readonly __brand: Name };

type Meters = Brand<number, "meters">;
type Seconds = Brand<number, "seconds">;
```

Идея такая:

```text
5 метров и 5 секунд — оба числа,
но складывать их нельзя
```

Это хороший пример того, что тип — не просто машинное представление. Машинно и метры, и секунды могут быть обычным `number`. Но смысл разный.

### 16. Типы как множества значений

Очень полезная ментальная модель:

```ts
boolean
```

это множество:

```text
true | false
```

Тип:

```ts
"left" | "right"
```

это множество из двух возможных значений:

```text
"left"
"right"
```

Тип:

```ts
never
```

это пустое множество. Значений такого типа не существует.

Тип:

```ts
unknown
```

это «какое-то значение, но мы пока не знаем какое».

Тип:

```ts
any
```

это «отключить проверку, я сам знаю, что делаю». Иногда полезно, часто опасно.

В TypeScript удобно думать именно так:

```ts
type A = "a" | "b" | "c";
type B = "b" | "c" | "d";

type Common = A & B;
// "b" | "c"
```

## Почему алгоритм часто описывают через последовательность, выбор и повторение

«Последовательность, выбор, повторение» — это не полный список всех понятий программирования.

Это минимальный набор структур управления потоком, из которых можно собрать вычисление в императивном стиле.

Они отвечают на вопрос:

```text
как движется выполнение программы?
```

### 1. Последовательность

```ts
a();
b();
c();
```

Сначала `a`, потом `b`, потом `c`.

### 2. Выбор

```ts
if (temperature > 100) {
  stop();
} else {
  continueWork();
}
```

Программа может пойти по одному пути или по другому.

Сюда относятся:

```text
if
else
switch
match
pattern matching
ternary operator
```

### 3. Повторение

```ts
while (hasEnergy()) {
  move();
}
```

Сюда относятся:

```text
while
for
for...of
do...while
recursion
```

Да, рекурсия тоже может играть роль повторения:

```ts
function factorial(n: number): number {
  if (n === 0) {
    return 1;
  }

  return n * factorial(n - 1);
}
```

Здесь нет `while`, но есть повторный вызов функции. По смыслу это повторение.

### А функции тогда где?

Функция — это не просто ещё один элемент рядом с `if` и `while`.

Функция — это в первую очередь абстракция и композиция.

Она позволяет взять кусок алгоритма:

```ts
const distance = Math.sqrt(
  (x2 - x1) ** 2 + (y2 - y1) ** 2
);
```

и дать ему имя:

```ts
function distance(x1: number, y1: number, x2: number, y2: number) {
  return Math.sqrt((x2 - x1) ** 2 + (y2 - y1) ** 2);
}
```

Теперь мы можем думать не о деталях, а о понятии:

```ts
if (distance(player.x, player.y, enemy.x, enemy.y) < 10) {
  attack();
}
```

Функция отвечает не на вопрос:

```text
куда пойдёт поток выполнения?
```

А на вопрос:

```text
как назвать и переиспользовать кусок смысла?
```

На машинном уровне функция тоже управляет потоком:

```text
прыгнуть в начало функции
сохранить адрес возврата
выполнить тело
вернуться назад
```

То есть функция — это тоже «прирученный GOTO с возвратом». Но в языках высокого уровня она важнее как единица смысла.

### Функции — это четвёртая базовая вещь?

Смотря в каком разговоре.

Если мы говорим о структурном программировании, то классическая тройка:

```text
последовательность
выбор
повторение
```

Это именно базовые конструкции потока управления.

А если мы говорим о построении больших программ, то да, функции абсолютно базовы:

```text
последовательность
выбор
повторение
абстракция
композиция
```

Для книги можно сказать так:

> Робот сказал неторопливо:  
> — У алгоритма есть три ноги: «сделай следующее», «выбери путь» и «повтори». Но у большой программы есть ещё руки: «дай этому имя» и «собери из маленьких частей большую». Эти руки называются функциями.

### Какие ещё элементы есть кроме трёх

Есть несколько уровней.

#### Уровень управления потоком

```text
sequence     — последовательность
selection    — выбор
iteration    — повторение
call/return  — вызов и возврат
exception    — аварийный выход
coroutine    — приостановка и продолжение
concurrency  — несколько потоков выполнения
```

Например исключение:

```ts
function readConfig() {
  if (!fileExists()) {
    throw new Error("No config");
  }
}
```

Это тоже управление потоком. Только необычное: не обычный `return`, а прыжок вверх по стеку вызовов до ближайшего `catch`.

```ts
try {
  readConfig();
} catch (error) {
  useDefaultConfig();
}
```

#### Уровень организации данных

```text
struct/object
array/list
map/dictionary
set
tree
graph
queue
stack
```

Это уже не «как идёт выполнение», а «как устроены данные».

#### Уровень абстракции

```text
function
module
class
interface
trait
generic
closure
```

Это способы строить большую систему из маленьких частей.

#### Уровень вычислительной модели

```text
imperative programming
functional programming
logic programming
dataflow programming
actor model
reactive programming
event-driven programming
```

Например frontend на React сильно живёт в модели:

```text
state -> render -> events -> new state -> render
```

Там не всегда удобно думать «сначала сделай A, потом B, потом C». Часто удобнее думать:

```text
вот состояние
вот как UI получается из состояния
вот какие события меняют состояние
```

### Почему всё-таки говорят про три вещи

Потому что на низком структурном уровне очень многое можно свести к этому:

```text
1. Выполни шаги по порядку.
2. Иногда выбирай одну из веток.
3. Иногда повторяй блок.
```

Например поиск максимума в массиве:

```ts
function max(numbers: number[]) {
  let current = numbers[0];

  for (const n of numbers) {
    if (n > current) {
      current = n;
    }
  }

  return current;
}
```

Тут есть всё:

```text
последовательность:
  создать current
  пройти по массиву
  вернуть current

повторение:
  for (const n of numbers)

выбор:
  if (n > current)
```

Функция `max` здесь — упаковка этого алгоритма под именем.

### Функция не заменяет выбор и повторение

Функция может содержать выбор:

```ts
function getPrice(user: User) {
  if (user.isPremium) {
    return 100;
  }

  return 150;
}
```

Функция может содержать повторение:

```ts
function sum(numbers: number[]) {
  let result = 0;

  for (const n of numbers) {
    result += n;
  }

  return result;
}
```

Функция может сама быть использована в последовательности:

```ts
const user = loadUser();
const price = getPrice(user);
sendInvoice(user, price);
```

То есть функция — это контейнер смысла, а внутри неё всё равно часто живут:

```text
последовательность
выбор
повторение
```

### В функциональном программировании повторение часто прячут

Вместо:

```ts
const result = [];

for (const user of users) {
  if (user.age >= 18) {
    result.push(user.name);
  }
}
```

пишут:

```ts
const result = users
  .filter(user => user.age >= 18)
  .map(user => user.name);
```

Тут вроде бы нет `if` и `for`.

Но они никуда не исчезли. Они спрятаны внутри `filter` и `map`.

```text
filter // повторение + выбор
map    // повторение + преобразование
```

Функциональное программирование часто не отменяет базовые элементы, а поднимает их на уровень выше.

### Минимальная карта

```text
Данные:
  примитивы
  структуры
  коллекции
  варианты/union
  функции как значения
  generic-типы

Поток выполнения:
  последовательность
  выбор
  повторение
  вызов/возврат
  исключения
  асинхронность/события
  параллельность

Абстракция:
  функции
  модули
  классы
  интерфейсы
  типы
  generic-и

Большие модели:
  императивная
  объектная
  функциональная
  логическая
  реактивная
  dataflow
  actor model
```

Самая короткая формула:

> Типы отвечают на вопрос: **что это за значения и что с ними можно делать?**  
> Управляющие конструкции отвечают на вопрос: **куда и когда идёт выполнение?**  
> Функции отвечают на вопрос: **как дать имя куску поведения и использовать его как строительный блок?**
