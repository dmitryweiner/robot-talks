# Консоль Linux: происхождение, аргументы, перенаправление и полезные утилиты

## Вопрос 1

**Как вообще возникла консоль (то что в Linux)? Это было отдельное устройство? Какие принципы работы консоли?**

## Ответ

Да, исторически консоль и терминал были **отдельными физическими устройствами**. В ранних компьютерах человек часто работал не за самим компьютером, а за устройством ввода-вывода: сначала это мог быть телетайп, затем видеотерминал с экраном и клавиатурой.

Важно различать несколько понятий:

- **консоль** — место управления компьютером;
- **терминал** — устройство или программа для текстового ввода-вывода;
- **TTY** — исторически телетайп, в Unix/Linux — абстракция терминального устройства;
- **shell** — программа, которая читает команды: `sh`, `bash`, `zsh`, `fish`;
- **эмулятор терминала** — окно вроде GNOME Terminal, Konsole, xterm, Alacritty.

Изначально терминал был довольно «тупым» устройством. Он не выполнял программы пользователя, а только:

1. принимал нажатия клавиш;
2. отправлял символы компьютеру;
3. получал символы от компьютера;
4. показывал их на экране или печатал на бумаге.

Схематично:

```text
человек → терминал → компьютер → терминал → человек
```

Unix создавался в мире, где один большой компьютер мог обслуживать много пользователей, а каждый пользователь подключался через свой терминал:

```text
[терминал Алисы] ─┐
[терминал Боба]  ├── [Unix-машина]
[терминал Кэрол] ─┘
```

Отсюда в Linux до сих пор существуют устройства вроде:

```text
/dev/tty1
/dev/tty2
/dev/pts/0
/dev/pts/1
```

Они не всегда соответствуют физическим устройствам. Часто это программные абстракции.

Когда ты открываешь окно терминала в Linux, происходит примерно следующее:

```text
[клавиатура]
   ↓
[эмулятор терминала]
   ↓
[псевдотерминал: /dev/pts/3]
   ↓
[shell: bash/zsh]
   ↓
[программы: ls, vim, ssh, git]
```

Терминал сам не знает, что такое `ls`, `cd` или `mkdir`. Он просто передаёт символы shell'у. Shell уже разбирает строку, запускает программы и соединяет их с вводом/выводом.

В Linux есть ещё **виртуальные консоли**, обычно доступные через:

```text
Ctrl + Alt + F1
Ctrl + Alt + F2
Ctrl + Alt + F3
...
```

Это текстовые интерфейсы системы, не зависящие от графического окружения. Если графическая оболочка сломалась, часто можно перейти туда и починить систему.

Одна из ключевых идей Unix: устройства представлены как файлы. Терминал тоже выглядит для программ как файлоподобный объект:

```text
/dev/tty
/dev/tty1
/dev/pts/0
```

У программы обычно есть три стандартных потока:

```text
stdin   — стандартный ввод
stdout  — стандартный вывод
stderr  — стандартный вывод ошибок
```

Когда программа пишет в `stdout`, терминал показывает текст. Когда программа читает из `stdin`, она получает то, что пользователь вводит с клавиатуры или что ей передали из другой программы.

Цвета, очистка экрана, перемещение курсора и прочие эффекты работают через управляющие escape-последовательности. Например, программа может отправить терминалу специальную последовательность символов, означающую «очистить экран» или «дальше пиши красным».

Главная идея консоли: **человек и программы обмениваются потоками текста**. Из-за этой простоты консоль пережила телетайпы, видеотерминалы, графические интерфейсы и до сих пор остаётся мощным способом управления системой.

## Вопрос 2

**Расскажи подробнее про аргументы программ, про перенаправление ввода/вывода, про то комбинирование разных команд**

## Ответ

Командная строка — это не просто место, куда пишут магические слова. Shell разбирает строку, выделяет имя программы, аргументы, перенаправления, пайпы и затем запускает процессы.

Например:

```bash
ls -l /home/dmitry
```

Shell видит примерно это:

```text
программа: ls
аргументы:
  -l
  /home/dmitry
```

С точки зрения программы аргументы — это просто массив строк. Для команды:

```bash
cp file.txt backup.txt
```

программа получает примерно:

```text
argv[0] = "cp"
argv[1] = "file.txt"
argv[2] = "backup.txt"
```

Смысл аргументов задаёт сама программа. Shell не знает, что `-l` означает «длинный формат», а `-n 10` означает «10 строк». Он только передаёт строки.

Аргументы бывают условно двух типов:

```bash
ls -la /etc
```

Здесь:

- `-la` — флаги, меняющие поведение;
- `/etc` — позиционный аргумент, то есть объект действия.

Кавычки нужны, чтобы передать один аргумент с пробелами:

```bash
mkdir "my folder"
grep "hello world" file.txt
```

Без кавычек shell разделил бы строку по пробелам.

У каждой консольной программы обычно есть три стандартных потока:

```text
0 — stdin   — ввод
1 — stdout  — обычный вывод
2 — stderr  — ошибки
```

Обычно `stdin` идёт с клавиатуры, а `stdout` и `stderr` — на экран. Но shell может это изменить.

### Перенаправление вывода

```bash
echo "hello" > file.txt
```

Это значит: запустить `echo`, но его обычный вывод отправить не на экран, а в файл `file.txt`.

```text
echo → file.txt
```

Оператор `>` перезаписывает файл. Оператор `>>` дописывает в конец:

```bash
echo "line 1" > log.txt
echo "line 2" >> log.txt
```

### Перенаправление ввода

```bash
wc -l < file.txt
```

Это значит: программа `wc -l` читает не с клавиатуры, а из файла.

```text
file.txt → wc -l → экран
```

Можно одновременно перенаправить ввод и вывод:

```bash
wc -l < app.log > result.txt
```

```text
app.log → wc -l → result.txt
```

### Перенаправление ошибок

Ошибки идут в отдельный поток `stderr`. Его можно перенаправить отдельно:

```bash
ls existing.txt missing.txt > out.txt 2> errors.txt
```

Обычный вывод попадёт в `out.txt`, ошибки — в `errors.txt`.

Чтобы отправить всё в один файл:

```bash
command > all.txt 2>&1
```

Есть ещё специальная «чёрная дыра»:

```bash
command > /dev/null 2>&1
```

Всё, что записывается в `/dev/null`, исчезает.

### Pipe: соединение команд

Оператор `|` соединяет `stdout` одной программы со `stdin` другой:

```bash
ls | wc -l
```

Схема:

```text
ls → wc -l → экран
```

`ls` выводит список файлов, `wc -l` считает строки.

Более выразительный пример:

```bash
grep "ERROR" app.log | wc -l
```

Смысл: найти строки с `ERROR` и посчитать их количество.

```text
app.log → grep → wc -l → экран
```

Команды можно соединять в длинные цепочки:

```bash
grep "ERROR" app.log | cut -d ' ' -f 1 | sort | uniq -c | sort -nr | head
```

Читается так:

1. взять строки с `ERROR`;
2. вытащить первый столбец;
3. отсортировать;
4. посчитать повторы;
5. отсортировать по числу повторов;
6. показать первые строки.

Это уже маленькая программа, собранная из простых утилит.

### Программы-фильтры

Многие Unix-утилиты устроены как фильтры:

```text
stdin → программа → stdout
```

Например:

- `grep` — фильтрует строки;
- `sort` — сортирует;
- `uniq` — убирает соседние дубликаты;
- `wc` — считает строки, слова, байты;
- `cut` — вырезает столбцы;
- `head` — показывает начало;
- `tail` — показывает конец;
- `sed` — редактирует поток текста;
- `awk` — обрабатывает строки и столбцы.

Пример классического паттерна:

```bash
sort names.txt | uniq -c
```

`sort` ставит одинаковые строки рядом, а `uniq -c` считает их.

### Подстановка команд

Иногда вывод одной команды нужен не как ввод другой, а как аргумент. Тогда используют `$(...)`:

```bash
mkdir "backup-$(date +%Y-%m-%d)"
```

Shell сначала выполнит `date +%Y-%m-%d`, а потом подставит результат в имя директории.

Разница:

```bash
cmd1 | cmd2
```

передаёт вывод `cmd1` во ввод `cmd2`.

```bash
cmd2 "$(cmd1)"
```

передаёт вывод `cmd1` как аргумент для `cmd2`.

### xargs

Иногда одна программа выдаёт список через `stdout`, а другая ждёт этот список как аргументы. Для этого есть `xargs`:

```bash
find . -name "*.tmp" | xargs rm
```

Схема:

```text
find → список файлов → xargs → rm file1 file2 file3
```

Более безопасный вариант для имён с пробелами:

```bash
find . -name "*.tmp" -print0 | xargs -0 rm
```

### tee

`tee` позволяет одновременно показать вывод на экране и записать его в файл:

```bash
ls -la | tee files.txt
```

```text
ls -la → tee → экран
              ↘ files.txt
```

### Условия выполнения

Shell умеет комбинировать команды не только через потоки, но и через коды возврата.

```bash
npm test && npm run build
```

Смысл: если тесты прошли успешно, запустить сборку.

```bash
npm test || echo "Tests failed"
```

Смысл: если тесты упали, вывести сообщение.

Главная философия Unix-консоли: маленькие программы должны хорошо соединяться друг с другом через текстовые потоки.

## Вопрос 3

**А в чём разница между перенаправлением ввода > и пайпом |, не могу понять?**

## Ответ

Тут важно поправить термин: `>` — это не перенаправление ввода, а **перенаправление вывода**.

Коротко:

```text
>  отправляет stdout программы в файл
<  берёт stdin программы из файла
|  соединяет stdout одной программы со stdin другой программы
```

Например:

```bash
echo hello > file.txt
```

означает:

```text
echo → file.txt
```

Вывод команды записывается в файл.

А вот:

```bash
echo hello | wc -c
```

означает:

```text
echo → wc -c → экран
```

Вывод `echo` передаётся другой программе, которая считает количество байтов/символов.

Сравнение:

```bash
ls > files.txt
```

Сохранить список файлов в файл.

```bash
ls | wc -l
```

Передать список файлов программе `wc -l`, чтобы она посчитала строки.

Pipeline можно представить как временную трубу без создания файла. Команда:

```bash
grep "ERROR" app.log | wc -l
```

похожа по смыслу на:

```bash
grep "ERROR" app.log > temp.txt
wc -l < temp.txt
```

Только без временного файла.

Разница между `>` и `<`:

```bash
command > file
```

«Положи вывод команды в файл».

```bash
command < file
```

«Возьми ввод команды из файла».

А `|`:

```bash
command1 | command2
```

«Положи вывод первой команды во ввод второй команды».

Самая короткая формула:

```text
>  в файл
<  из файла
|  в другую программу
```

## Вопрос 4

**Какие ещё полезные консольные утилиты существуют?**

## Ответ

Консольные утилиты удобно воспринимать как набор маленьких инструментов. Одни работают с файлами, другие ищут текст, третьи смотрят процессы, четвёртые помогают анализировать данные.

### Навигация и просмотр файлов

```bash
pwd
ls -la
cd /path/to/dir
cd ..
cd -
```

- `pwd` показывает текущую директорию;
- `ls -la` показывает файлы подробно, включая скрытые;
- `cd` меняет директорию;
- `cd -` возвращает в предыдущую директорию.

Для просмотра файлов:

```bash
less file.txt
head file.txt
tail file.txt
tail -f app.log
```

- `less` удобен для длинных файлов;
- `head` показывает начало;
- `tail` показывает конец;
- `tail -f` следит за логом в реальном времени.

### Работа с файлами

```bash
cp source.txt target.txt
mv old.txt new.txt
rm file.txt
mkdir new-dir
touch file.txt
```

- `cp` копирует;
- `mv` перемещает или переименовывает;
- `rm` удаляет;
- `mkdir` создаёт директорию;
- `touch` создаёт пустой файл или обновляет время изменения.

Опасная, но часто используемая команда:

```bash
rm -rf dir
```

Она удаляет директорию рекурсивно и без вопросов. С ней легко ошибиться.

Полезные команды для информации о файлах:

```bash
stat file.txt
file something
ln -s real/path link-name
```

- `stat` показывает метаданные файла;
- `file` определяет тип файла по содержимому;
- `ln -s` создаёт символическую ссылку.

### Поиск файлов

```bash
find . -name "*.js"
find . -type f -mtime -7
find . -name "*.tmp" -delete
```

- найти `.js` файлы;
- найти файлы, изменённые за последние 7 дней;
- найти и удалить временные файлы.

Современная удобная альтернатива:

```bash
fd pattern
```

`fd` обычно приятнее `find`, но часто требует установки.

### Поиск по содержимому

```bash
grep "ERROR" app.log
grep -rn "TODO" .
grep -v "DEBUG" app.log
```

- `grep` ищет строки;
- `-r` — рекурсивно;
- `-n` — с номерами строк;
- `-v` — показать строки, которые не совпали.

Современная альтернатива:

```bash
rg "useEffect"
rg "TODO|FIXME" src/
```

`rg`, он же ripgrep, очень удобен для поиска по проектам.

### Текстовые фильтры

```bash
wc -l file.txt
sort names.txt
sort names.txt | uniq -c
cut -d ',' -f 1 data.csv
```

- `wc` считает строки, слова, байты;
- `sort` сортирует;
- `uniq` убирает соседние дубликаты;
- `cut` вырезает столбцы.

Более мощные инструменты:

```bash
sed 's/old/new/g' file.txt
awk '{print $1, $3}' file.txt
```

- `sed` редактирует поток текста;
- `awk` удобен для обработки строк и столбцов.

Пример с `awk`:

```bash
awk '{sum += $1} END {print sum}' numbers.txt
```

Суммирует первый столбец.

### JSON, YAML и структурированные данные

Для JSON:

```bash
jq '.scripts' package.json
jq '.items[] | .title' file.json
```

`jq` — очень полезная утилита для работы с JSON.

Для YAML часто используют:

```bash
yq '.services' docker-compose.yml
```

### Комбинирование команд

```bash
command1 | command2
command > out.txt
command >> out.txt
command 2> errors.txt
command | tee out.txt
```

- `|` передаёт вывод одной команды во ввод другой;
- `>` записывает вывод в файл;
- `>>` дописывает в конец;
- `2>` перенаправляет ошибки;
- `tee` одновременно показывает вывод и сохраняет его.

### Процессы и ресурсы

```bash
ps aux
ps aux | grep node
top
htop
kill PID
pgrep node
pkill node
```

- `ps` показывает процессы;
- `top`/`htop` показывают нагрузку в реальном времени;
- `kill` отправляет процессу сигнал;
- `pgrep` и `pkill` ищут/завершают процессы по имени.

### Диски и память

```bash
df -h
du -sh *
free -h
lsblk
```

- `df -h` показывает свободное место;
- `du -sh *` показывает размеры файлов и папок;
- `free -h` показывает память;
- `lsblk` показывает диски и разделы.

### Сеть

```bash
ip a
ip route
ping 1.1.1.1
curl -I https://example.com
ssh user@host
scp file.txt user@host:/path/
rsync -av dir/ user@host:/path/
```

- `ip a` показывает сетевые интерфейсы;
- `ip route` показывает маршруты;
- `ping` проверяет связь;
- `curl` делает HTTP-запросы;
- `ssh` подключается к удалённой машине;
- `scp` копирует файлы;
- `rsync` синхронизирует файлы и копирует только изменения.

### Архивы

```bash
tar -czf archive.tar.gz folder/
tar -xzf archive.tar.gz
zip -r archive.zip folder/
unzip archive.zip
```

- `tar -czf` создаёт `.tar.gz`;
- `tar -xzf` распаковывает;
- `zip` и `unzip` работают с zip-архивами.

### Права и пользователи

```bash
whoami
id
chmod +x script.sh
chmod 755 script.sh
chown user:group file.txt
sudo command
```

- `whoami` показывает текущего пользователя;
- `id` показывает UID, GID и группы;
- `chmod` меняет права;
- `chown` меняет владельца;
- `sudo` запускает команду с повышенными правами.

### Система и логи

```bash
uname -a
cat /etc/os-release
dmesg
journalctl -xe
journalctl -u nginx
journalctl -f
```

- `uname` показывает информацию о ядре;
- `/etc/os-release` — информация о дистрибутиве;
- `dmesg` — сообщения ядра;
- `journalctl` — системные логи в systemd-системах.

### Для разработчика

```bash
git status
git diff
git log --oneline --graph --decorate
python3 -m http.server 8000
which node
command -v node
time npm test
```

- `git` — контроль версий;
- `python3 -m http.server` быстро поднимает HTTP-сервер;
- `which`/`command -v` показывают, откуда запускается команда;
- `time` измеряет время выполнения.

### Современные утилиты, которые стоит поставить

```bash
rg      # быстрый grep для проектов
fd      # удобный find
jq      # обработка JSON
fzf     # интерактивный fuzzy finder
bat     # cat с подсветкой синтаксиса
eza     # современный ls
zoxide  # умный cd
tldr    # короткие практические примеры команд
ncdu    # удобный анализ занятого места
btop    # красивый монитор ресурсов
```

Главная идея: консольная утилита ценна не только сама по себе, а тем, что её можно встроить в цепочку.

Например:

```bash
find . -name "*.log" -type f \
  | xargs grep -h "ERROR" \
  | cut -d " " -f 1 \
  | sort \
  | uniq -c \
  | sort -nr \
  | head
```
