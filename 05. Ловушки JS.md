# Глава. Ложные пустоты

Дождь шёл уже третий день. Он не был сильным — скорее, бесконечным: тонкие струи стекали по мутным стёклам бывшего учебного корпуса, собирались на подоконниках, капали в алюминиевые лотки, которые робот расставил ещё утром.

В комнате было сухо. Почти тепло. На столе горела лампа с матовым плафоном, питавшаяся от аккумуляторного блока. За стеной, в серверной, тихо работал вентилятор: не так ровно, как когда-то, а с лёгким утомлённым подвыванием, будто сам воздух вспоминал, что ему положено двигаться по правилам.

Мальчик сидел на старом лабораторном стуле, поджав под себя одну ногу. Перед ним был терминал. Не настоящий терминал, как говорил робот, а «достаточно настоящий»: экран, клавиатура, маленький одноплатный компьютер, старый браузер и локальная среда для JavaScript. На краю стола лежала вскрытая упаковка сухого пайка. Мальчик время от времени отламывал кусочек галеты и забывал жевать, когда код начинал вести себя странно.

Робот стоял рядом с открытой панелью в груди. Внутри ровно светились индикаторы. Он проверял напряжение на одном из аккумуляторных модулей и одновременно смотрел на экран.

— Я нашёл странную ошибку, — сказал мальчик. — Не страшную, но противную. Такую, которая выглядит как ловушка.

— Покажи.

Мальчик повернул экран. На нём был короткий фрагмент:

```js
const [result] = getResult() || [];
```

— Я думал, это аккуратно, — сказал он. — Если результата нет, взять пустой массив. Но потом `getResult()` вернул ноль, и программа упала. Потому что ноль нельзя распаковать как массив. А если бы я написал `?? []`, всё бы работало. Это ведь не ошибка алгоритма. Это как будто я наступил на доску, которая выглядела крепкой, а под ней была яма.

Робот закрыл панель в груди. Защёлка мягко щёлкнула.

— Хорошее описание. В JavaScript много таких досок. Они были сделаны не для того, чтобы вредить, а потому что язык рос быстро, менялся, сохранял обратную совместимость и пытался быть удобным для коротких программ. Но короткая программа и надёжная программа — не всегда одно и то же.

— Значит, надо просто не писать коротко?

— Не совсем. Надо понимать, какую именно пустоту ты имеешь в виду.

Мальчик нахмурился.

— Пустоту?

— В JavaScript есть несколько разных состояний, которые человек часто смешивает. Вот, например: значения нет. Значение есть, но оно равно нулю. Значение есть, но это пустая строка. Значение есть, но это `false`. Значение есть, но оно не число. Для человека всё это иногда звучит как «ничего». Для программы это разные вещи.

Робот взял с полки тонкую пластиковую пластину и положил её рядом с клавиатурой.

— Представь склад после людей. На одной полке нет ящика. На другой стоит пустой ящик. На третьей лежит ящик с надписью «ноль деталей». На четвёртой — ящик с выключенным переключателем. Если ты скажешь роботу-сборщику: «Если там не полезное значение, бери запасной ящик», он может выбросить и пустой ящик, и нулевой счётчик, и выключенный флаг. Иногда это правильно. Иногда это катастрофа.

— Это `||`?

— Да. Оператор `||` берёт правую сторону не только когда слева `null` или `undefined`. Он берёт правую сторону при любом так называемом falsy-значении.

Робот напечатал:

```js
false
0
""
NaN
null
undefined
```

— Все они для `||` выглядят как «нет».

Мальчик кивнул.

— А `??`?

— `??` гораздо уже. Он говорит: «используй запасное значение только если слева действительно нет значения — `null` или `undefined`».

Он изменил код:

```js
const [result] = getResult() ?? [];
```

— Это лучше, если `getResult()` может вернуть пустую строку, ноль или `false`, и эти значения важны.

— Но мой пример всё равно странный, да? — спросил мальчик. — Если функция может вернуть ноль, то почему я пытаюсь распаковать её как массив?

— Именно. Ты поймал не только ошибку оператора, но и запах плохого договора между частями программы. Если функция называется `getResult`, непонятно, что она возвращает: массив, число, ничего? Надёжнее сделать контракт яснее. Например:

```js
const value = getResult();

const [result] = Array.isArray(value) ? value : [];
```

— Это длиннее.

— Да. Но оно говорит правду. Иногда длинная запись — это не лишние слова, а табличка на двери: «за этой дверью шахта».

Мальчик посмотрел на код и медленно сказал:

— То есть первая ловушка: не все пустые значения одинаковые.

— Верно.

За окном вода стекала с рамы длинной дрожащей линией. Где-то внизу, в бывшей столовой, с глухим звуком упала очередная капля в металлический таз.

— А таких ловушек много? — спросил мальчик.

— Достаточно, чтобы уметь их узнавать. Давай разберём не все сразу, а по семействам. Тогда они перестанут быть набором суеверий.

## Ловушка первая: ложные пустоты

Робот открыл новый файл и написал:

```js
function formatCount(count) {
  return count || "—";
}
```

— Что вернёт `formatCount(5)`?

— Пять.

— А `formatCount(null)`?

— Черту.

— А `formatCount(0)`?

Мальчик замолчал.

— Тоже черту, — сказал он через секунду. — Хотя ноль — это нормальное количество.

— Да.

Робот заменил строку:

```js
function formatCount(count) {
  return count ?? "—";
}
```

— Теперь черта появится только если значение отсутствует: `null` или `undefined`.

— Значит, для чисел почти всегда лучше `??`.

— Для дефолтов — часто да. Особенно если ноль допустим. То же самое с настройками:

```js
const enabled = config.enabled || true;
```

— Ой, — сказал мальчик. — Если `enabled` равен `false`, оно станет `true`.

— Да. А пользователь явно хотел выключить.

```js
const enabled = config.enabled ?? true;
```

— Так программа уважает `false`.

Мальчик откусил галету и наконец прожевал.

— Это похоже на ремонт старого щита. Если рубильник в положении «выключено», это не значит, что рубильника нет.

— Хорошая аналогия, — сказал робот. — `false` — это положение рубильника. `undefined` — это отсутствие рубильника.

## Ловушка вторая: распаковка того, чего нет

Робот написал другой пример:

```js
function renderUser({ name }) {
  console.log(name);
}

renderUser(undefined);
```

— Упадёт, — сказал мальчик. — Потому что нельзя достать `name` из `undefined`.

— Правильно. Это destructuring. Очень удобная конструкция, но она предполагает, что справа есть объект или массив.

Он написал:

```js
function renderUser(user) {
  const { name } = user ?? {};
  console.log(name);
}
```

— Здесь мы говорим: если пользователя нет, возьми пустой объект.

— А можно так? — мальчик наклонился к клавиатуре.

```js
function renderUser(user = {}) {
  const { name } = user;
}
```

— Можно, но это спасёт только от `undefined`. Если передать `null`, параметр по умолчанию не сработает.

— Почему?

— Потому что параметры по умолчанию в JavaScript применяются только когда аргумент именно `undefined`. `null` считается переданным значением. Как пустая корзина: она пустая, но она есть.

Мальчик помолчал, потом спросил:

— А с вложенными объектами ещё хуже?

— Да.

Робот написал:

```js
const {
  settings: {
    theme
  }
} = user;
```

— Этот код требует, чтобы существовал `user`, внутри него `settings`, а внутри `settings` — `theme`. Если `settings` нет, он упадёт.

— Лучше так?

```js
const theme = user?.settings?.theme ?? "light";
```

— Да. Здесь optional chaining идёт шаг за шагом. Но и у него есть тонкость.

Робот добавил:

```js
const name = (user?.profile).name;
```

— Это безопасно? — спросил он.

Мальчик сначала кивнул, потом прищурился.

— Скобки… `user?.profile` может вернуть `undefined`, а потом мы делаем `.name` уже у `undefined`.

— Верно. Цепочка безопасности оборвалась скобками.

Правильно:

```js
const name = user?.profile?.name;
```

— Optional chaining похож на верёвку над провалом, — сказал робот. — Она должна тянуться через весь опасный участок. Если ты сделал безопасным только первый шаг, второй всё равно может быть в пустоту.

## Ловушка третья: красивые сокращения, которые едят данные

Робот открыл массив:

```js
const values = [0, 1, 2, null, undefined, 3];

const clean = values.filter(Boolean);
```

— Это я видел, — сказал мальчик. — Так удаляют пустые значения.

— Какие именно пустые?

Мальчик посмотрел внимательнее.

— `null`, `undefined`… и ноль тоже.

— Да. Результат будет:

```js
[1, 2, 3]
```

— Но если ноль был показанием датчика, мы его потеряли.

— Именно. Представь водомер на старой насосной станции. Ноль литров в минуту — это не отсутствие датчика. Это важное показание.

Робот написал:

```js
const clean = values.filter(value => value != null);
```

— Это убирает только `null` и `undefined`.

— Но ты использовал `!=`, — заметил мальчик. — Ты же говорил, что лучше `===`.

— Хорошее замечание. Обычно лучше `===`. Но `value != null` — распространённое осознанное сокращение для:

```js
value !== null && value !== undefined
```

— То есть оно допустимо, если команда договорилась?

— Да. Или можно писать длинно, зато совсем явно.

---

## Ловушка четвёртая: функция получила больше аргументов, чем ты думал

— Есть одна старая ловушка, — сказал робот. — Её любят показывать детям, взрослым и программистам, потому что все трое сначала не верят.

Он написал:

```js
["10", "10", "10"].map(parseInt);
```

— Должно быть `[10, 10, 10]`.

— А будет:

```js
[10, NaN, 2]
```

Мальчик нахмурился.

— Почему два?

— `map` передаёт функции не только элемент. Он передаёт элемент, индекс и сам массив:

```js
(value, index, array)
```

А `parseInt` принимает второй аргумент как основание системы счисления.

Робот разложил:

```js
parseInt("10", 0); // 10
parseInt("10", 1); // NaN
parseInt("10", 2); // 2
```

— То есть индекс случайно стал основанием.

— Да. Программа не знает, что ты «имел в виду». Она только передаёт аргументы.

Правильно:

```js
["10", "10", "10"].map(value => parseInt(value, 10));
```

Или:

```js
["10", "10", "10"].map(Number);
```

— Но `Number` и `parseInt` не одно и то же, — добавил робот.

```js
Number("10px");   // NaN
parseInt("10px"); // 10
```

— `parseInt` как будто читает число до первого мусора, — сказал мальчик.

— Да. Иногда это нужно. Иногда это опасно.

## Ловушка пятая: массивы, которые не совсем массивы

Робот написал:

```js
Array(5).map((_, i) => i);
```

— Ноль, один, два, три, четыре? — предположил мальчик уже осторожно.

— Нет. Получится массив из пяти пустых мест. `map` не проходит по дыркам.

— Дыркам?

— В JavaScript массив может иметь длину, но не иметь значения в некоторых позициях. Это не то же самое, что `undefined`.

Робот показал:

```js
const arr = [1, , 3];

arr.length; // 3
```

— Между единицей и тройкой есть отверстие. Некоторые методы его пропускают, некоторые ведут себя иначе.

— Это ужасно.

— Это исторически полезная гибкость, которая легко становится ужасом. Чтобы создать массив чисел от нуля до четырёх, лучше:

```js
Array.from({ length: 5 }, (_, i) => i);
```

— А что насчёт `fill`? — спросил мальчик. — Я видел такое:

```js
Array(3).fill({ selected: false });
```

— Ещё одна ловушка. Все три элемента будут ссылаться на один и тот же объект.

Робот дописал:

```js
const rows = Array(3).fill({ selected: false });

rows[0].selected = true;

console.log(rows);
```

— Все станут `selected: true`, — догадался мальчик.

— Да. Потому что объект один. Ты положил один и тот же ящик на три полки, а не сделал три ящика.

Правильно:

```js
const rows = Array.from(
  { length: 3 },
  () => ({ selected: false })
);
```

— Каждый вызов функции создаёт новый объект.

Мальчик постучал пальцем по столу.

— Значит, для примитивов `fill` нормально, а для объектов — опасно.

— Хорошая практическая формулировка.

## Ловушка шестая: методы, которые меняют старое

Робот набрал:

```js
const original = [3, 1, 2];
const sorted = original.sort((a, b) => a - b);
```

— `sorted` будет `[1, 2, 3]`.

— А `original`?

Мальчик ответил не сразу.

— Тоже?

— Да. `sort()` мутирует массив. Как и `reverse()`, `splice()`, `push()`, `pop()`, `shift()`, `unshift()` и некоторые другие.

— То есть если старый массив ещё где-то используется, я могу сломать другое место.

— Да. В больших программах это похоже на старый водопровод: ты повернул вентиль в одном подвале, а вода исчезла в другом здании.

Лучше:

```js
const sorted = [...original].sort((a, b) => a - b);
```

— Мы сначала делаем копию, потом сортируем копию.

— А ещё есть новые методы, — добавил робот.

```js
const sorted = original.toSorted((a, b) => a - b);
const reversed = original.toReversed();
const changed = original.with(0, "new value");
```

— Они возвращают новый массив и не трогают старый.

— Это особенно важно в React? — спросил мальчик.

— Да. React часто определяет, изменились ли данные, по ссылке. Если ты мутировал старый массив, программа может не понять, что надо перерисовать интерфейс. Или наоборот — где-то изменится то, что не должно было меняться.

## Ловушка с сортировкой чисел

— И раз мы уже говорим о `sort`, у него есть ещё одна особенность.

```js
[1, 2, 10, 20].sort();
```

— Будет `[1, 2, 10, 20]`?

— Нет.

```js
[1, 10, 2, 20]
```

— Почему?

— По умолчанию `sort()` сравнивает элементы как строки. `"10"` идёт раньше `"2"`.

— Как каталожные номера в архиве, — сказал мальчик. — Если сортировать как текст, `A10` может идти перед `A2`.

— Именно. Для чисел нужно:

```js
[1, 2, 10, 20].sort((a, b) => a - b);
```

## Ловушка с циклами

Робот написал:

```js
const arr = ["a", "b", "c"];

for (const item in arr) {
  console.log(item);
}
```

— Выведет буквы?

— Нет. `for...in` проходит по ключам. Для массива это индексы: `"0"`, `"1"`, `"2"`.

Правильно:

```js
for (const item of arr) {
  console.log(item);
}
```

— `for...in` — по ключам объекта. `for...of` — по значениям iterable.

— Iterable — это то, по чему можно идти как по последовательности?

— Да. Массив, строка, Map, Set и многое другое.

## Ловушка с `this`

Дождь за окном стал тише. Робот на секунду повернул голову, прислушиваясь к звуку на крыше, затем продолжил:

— Теперь одна из самых старых и самых важных тем: `this`.

Он написал:

```js
const user = {
  name: "Alice",
  getName() {
    return this.name;
  }
};

const getName = user.getName;

getName();
```

— Должно вернуть `"Alice"`? — спросил мальчик, но уже без уверенности.

— Нет. Метод потерял объект, через который его вызвали. `this` в JavaScript определяется не тем, где функция написана, а тем, как она вызвана.

Работает:

```js
user.getName();
```

Не работает как метод:

```js
const fn = user.getName;
fn();
```

— То есть функция оторвалась от владельца.

— Да. Как инструмент, снятый со станка. Сам инструмент остался, но больше не знает, к какому станку он был прикручен.

Можно привязать:

```js
const getName = user.getName.bind(user);
```

Или вызвать через обёртку:

```js
const getName = () => user.getName();
```

— А стрелочные функции? Они же часто помогают.

— Они помогают, но не как методы объекта.

Робот написал:

```js
const user = {
  name: "Alice",
  getName: () => this.name
};
```

— Здесь `this` не будет `user`. Стрелочная функция не имеет своего `this`; она берёт его из внешней области.

— Значит, для методов объекта лучше обычная запись?

```js
const user = {
  name: "Alice",
  getName() {
    return this.name;
  }
};
```

— Да. А стрелки хороши для callback-функций, где нужно сохранить внешний `this` или просто коротко передать поведение.

## Ловушка с `return`

Робот посмотрел на мальчика.

— Вот эта ошибка особенно обидная.

Он написал:

```js
function getUser() {
  return
  {
    name: "Alice"
  };
}
```

Мальчик засмеялся тихо, почти беззвучно.

— Я знаю. Вернёт объект?

— Нет. Вернёт `undefined`.

— Из-за переноса строки?

— Да. JavaScript автоматически вставляет точку с запятой после `return`.

Как будто программа видит:

```js
function getUser() {
  return;
  {
    name: "Alice"
  }
}
```

Правильно:

```js
function getUser() {
  return {
    name: "Alice"
  };
}
```

— Поэтому в JSX часто пишут:

```jsx
return (
  <Component />
);
```

— Именно. Скобка после `return` говорит: выражение продолжается.

## Ловушка с промисами

Мальчик открыл другой файл.

— А вот это меня тоже сбивало. Я написал:

```js
items.forEach(async item => {
  await save(item);
});

console.log("done");
```

— И `done` появился раньше, чем всё сохранилось.

— Да.

— Потому что `forEach` не ждёт промисы. Он просто вызывает функцию для каждого элемента. То, что функция `async`, не меняет поведение самого `forEach`.

Если нужно последовательно:

```js
for (const item of items) {
  await save(item);
}

console.log("done");
```

Если можно параллельно:

```js
await Promise.all(items.map(item => save(item)));

console.log("done");
```

— А если я написал `map(async ...)`, то получаю массив промисов?

— Да.

```js
const results = items.map(async item => {
  return await process(item);
});
```

— `results` здесь не результаты, а `Promise[]`.

Правильно:

```js
const results = await Promise.all(
  items.map(item => process(item))
);
```

Мальчик кивнул.

— А `try/catch`?

Робот написал:

```js
try {
  doAsyncThing();
} catch (error) {
  console.error(error);
}
```

— Если `doAsyncThing` возвращает промис, ошибка не будет поймана этим `catch`, если ты не сделал `await`.

Правильно:

```js
try {
  await doAsyncThing();
} catch (error) {
  console.error(error);
}
```

— Значит, промис — это как отложенная работа. Если я не дождался рабочего, я не могу поймать ящик, который он уронит потом.

— Точная аналогия.

## Ловушка с `Promise.all`

— А если один из промисов падает? — спросил мальчик.

```js
await Promise.all([
  fetchUser(),
  fetchPosts(),
  fetchSettings()
]);
```

— Тогда весь `Promise.all` завершается ошибкой. Иногда это нужно: если без одного результата всё бессмысленно. Но если ты хочешь собрать всё, что получилось, используй:

```js
const results = await Promise.allSettled([
  fetchUser(),
  fetchPosts(),
  fetchSettings()
]);
```

— Там у каждого результата будет статус?

— Да: `fulfilled` или `rejected`.

## Ловушка с числами и строками

Робот написал:

```js
1 + 2
"1" + 2
1 + "2"
```

— Первое — три, остальные — строки.

— Да. Оператор `+` одновременно сложение и конкатенация. Если в вычисления попала строка, результат может стать строкой.

```js
[1, 2, "3", 4].reduce((sum, value) => sum + value);
```

— Что получится?

Мальчик прикинул.

— Сначала `1 + 2 = 3`, потом `3 + "3" = "33"`, потом `"33" + 4 = "334"`.

— Верно.

Лучше явно приводить и задавать начальное значение:

```js
const total = values.reduce(
  (sum, value) => sum + Number(value),
  0
);
```

— Но если там `"10px"`?

— `Number("10px")` даст `NaN`. Это может быть хорошо: мы узнали, что вход плохой. А `parseInt("10px", 10)` дал бы `10`, что иногда удобно, а иногда скрывает ошибку.

## Ловушка с `reduce`

— Кстати, о `reduce`, — продолжил робот.

```js
const sum = numbers.reduce((acc, x) => acc + x);
```

— На пустом массиве упадёт?

— Да. Потому что нет первого значения, с которого можно начать. Лучше почти всегда писать начальное значение:

```js
const sum = numbers.reduce((acc, x) => acc + x, 0);
```

Для объектов:

```js
const byId = users.reduce((acc, user) => {
  acc[user.id] = user;
  return acc;
}, {});
```

— Начальное значение — как пустая коробка перед сортировкой деталей, — сказал мальчик. — Без неё первый предмет приходится назначать коробкой, и если предметов нет, всё ломается.

— Именно.

## Ловушка с `NaN`

Робот написал:

```js
NaN === NaN
```

— Ложь, — сказал мальчик. — Я уже видел. Но до сих пор неприятно.

— `NaN` означает «не число», и исторически оно не равно даже самому себе. Проверять нужно так:

```js
Number.isNaN(value)
```

Не так:

```js
value === NaN
```

И лучше не так:

```js
isNaN(value)
```

— Почему?

— Глобальный `isNaN` сначала пытается привести значение к числу.

```js
isNaN("hello");        // true
Number.isNaN("hello"); // false
```

— То есть `Number.isNaN` спрашивает: «это действительно значение NaN?», а `isNaN` спрашивает: «станет ли это NaN, если я попробую сделать из него число?»

— Да.

---

## Ловушка с объектами

Робот взял со стола маленький металлический винт и положил его рядом с другим, почти таким же.

— Эти два винта похожи. Но это два разных винта.

На экране появилось:

```js
{} === {};
[] === [];
```

— Оба `false`.

— Потому что сравниваются ссылки, — сказал мальчик.

— Да. Объекты и массивы сравниваются не по содержимому, а по тому, один ли это объект в памяти.

То же с копированием:

```js
const user = {
  name: "Alice",
  settings: {
    theme: "dark"
  }
};

const copy = { ...user };

copy.settings.theme = "light";
```

— И у старого пользователя тема тоже станет `"light"`?

— Да. Spread делает поверхностную копию. Вложенный объект остаётся тем же.

Лучше:

```js
const copy = {
  ...user,
  settings: {
    ...user.settings,
    theme: "light"
  }
};
```

— Каждый уровень, который меняешь, надо копировать отдельно.

— Да. Это один из главных принципов аккуратной работы с неизменяемыми данными.

## Ловушка с `const`

Мальчик сам написал:

```js
const user = {
  name: "Alice"
};

user.name = "Bob";
```

— Это разрешено?

— Да. `const` запрещает переназначить переменную, но не замораживает объект.

Нельзя:

```js
user = {};
```

Можно:

```js
user.name = "Bob";
```

— То есть `const` говорит: «ярлык всегда привязан к этому ящику», но не говорит: «ящик нельзя открыть и переложить внутри детали».

— Отлично сказано.

## Ловушка с JSON

Робот написал:

```js
JSON.stringify({
  a: undefined,
  b: () => {},
  c: Symbol("x"),
  d: 123
});
```

— Что останется?

— Только `d`?

— Да.

```json
{"d":123}
```

— JSON — не универсальный способ сохранить всё. Он теряет функции, `undefined`, символы. Даты превращает в строки. `BigInt` вообще может уронить сериализацию.

— А `JSON.parse(JSON.stringify(obj))` для копирования?

— Опасная привычка. Она иногда работает для простых данных, но ломает типы: `Date`, `Map`, `Set`, `NaN`, `Infinity`, классы.

Лучше, если подходит:

```js
const copy = structuredClone(obj);
```

— Но и он не копирует всё подряд. Для важных данных лучше явно понимать, что ты копируешь.

## Ловушка с датами

Робот ненадолго открыл старую карту расписаний поездов. На ней ещё были видны линии, ведущие в города, где давно никто не ждал поездов.

— Даты — отдельное болото. Например:

```js
new Date("2026-01-01")
```

— Это первое января?

— Да, но интерпретация связана с UTC. А вот:

```js
new Date(2026, 0, 1)
```

— Месяц ноль?

— Январь. В старом `Date` месяцы считаются с нуля.

— Да. Один из богов старого мира.

## Ловушка с React и нулём

— Ты работаешь с интерфейсами, — сказал робот. — Значит, вот ещё одна частая ловушка:

```jsx
{items.length && <List items={items} />}
```

— Если длина ноль, React покажет `0`.

— Да. Потому что выражение `items.length && ...` возвращает сам ноль, а React умеет его отрисовать.

Лучше:

```jsx
{items.length > 0 && <List items={items} />}
```

— То есть опять надо явно сказать, что проверяю не «истинность», а «длина больше нуля».

— Именно. Это почти вся сегодняшняя глава в одной фразе.

## Ловушка с событиями

Робот взял старую компьютерную мышь, давно уже не подключённую, и положил рядом с клавиатурой.

— В браузере есть ещё такая разница:

```js
event.target
event.currentTarget
```

— `target` — где событие началось, `currentTarget` — где висит обработчик?

— Да. Если ты повесил обработчик на кнопку, но внутри кнопки есть `span`, то `event.target` может быть этим `span`.

```js
button.addEventListener("click", event => {
  event.target.disabled = true;
});
```

— Может попытаться выключить `span`.

— Верно. Надёжнее:

```js
button.addEventListener("click", event => {
  event.currentTarget.disabled = true;
});
```

## Ловушка с обработчиками

— И ещё одна:

```js
element.addEventListener("click", () => {
  console.log("click");
});

element.removeEventListener("click", () => {
  console.log("click");
});
```

— Не удалит?

— Не удалит. Это две разные функции. Они выглядят одинаково, но это два разных объекта.

Правильно:

```js
function handleClick() {
  console.log("click");
}

element.addEventListener("click", handleClick);
element.removeEventListener("click", handleClick);
```

— Как с двумя одинаковыми винтами, — сказал мальчик. — Внешне похожи, но один не заменяет другой, если механизм помнит именно первый.

— Да.

## Ловушка с регулярными выражениями

Робот написал:

```js
const regex = /foo/g;

regex.test("foo"); // true
regex.test("foo"); // false
regex.test("foo"); // true
```

Мальчик устало посмотрел на экран.

— Почему?

— Флаг `g` делает регулярное выражение состоянием. Оно запоминает `lastIndex`, позицию следующего поиска.

— То есть `test` меняет regex?

— В этом случае да. Для простой проверки лучше без `g`:

```js
/foo/.test("foo");
```

Или сбрасывать:

```js
regex.lastIndex = 0;
regex.test(value);
```

## Как с этим бороться

Мальчик отодвинул клавиатуру.

— Всё это похоже на минное поле. Как вообще писать и не бояться каждого символа?

— Не надо бояться. Надо строить защитные привычки.

Робот поднял один палец.

— Первое: использовать TypeScript в строгом режиме, если проект позволяет.

Он написал:

```json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "noFallthroughCasesInSwitch": true
  }
}
```

— `strict` включает много проверок. `noUncheckedIndexedAccess` напоминает, что `arr[0]` может быть `undefined`. `exactOptionalPropertyTypes` делает опциональные поля честнее. `noFallthroughCasesInSwitch` защищает от забытых `break`.

— А ESLint?

— Второе. Линтер — это как маленький сторожевой робот, который ходит по коду и говорит: «здесь ты, вероятно, забыл закрыть люк».

```json
{
  "rules": {
    "eqeqeq": ["error", "always", { "null": "ignore" }],
    "no-fallthrough": "error",
    "no-array-constructor": "error",
    "no-throw-literal": "error",
    "radix": "error"
  }
}
```

Для TypeScript полезны:

```json
{
  "rules": {
    "@typescript-eslint/no-floating-promises": "error",
    "@typescript-eslint/await-thenable": "error",
    "@typescript-eslint/no-misused-promises": "error",
    "@typescript-eslint/prefer-nullish-coalescing": "warn",
    "@typescript-eslint/prefer-optional-chain": "warn",
    "@typescript-eslint/strict-boolean-expressions": "warn"
  }
}
```

— `strict-boolean-expressions` — это где нельзя просто писать `if (value)`?

— Да. Оно заставляет уточнять:

```js
if (value != null) {}
if (value.length > 0) {}
if (value === true) {}
if (Number.isFinite(value)) {}
if (Array.isArray(value)) {}
```

— Сначала раздражает?

— Очень часто. Потом дисциплинирует.

Робот поднял второй палец.

— Третье: нормализовать данные на входе.

```js
function normalizeConfig(config) {
  return {
    enabled: config?.enabled ?? true,
    title: config?.title ?? "Untitled",
    items: Array.isArray(config?.items) ? config.items : []
  };
}
```

— То есть не размазывать проверки по всей программе.

— Да. На границе ты превращаешь грязные, неизвестные данные в понятную форму. Дальше код живёт спокойнее.

Робот поднял третий палец.

— Четвёртое: не писать слишком умные выражения.

Он набрал:

```js
const result = data?.items?.filter(Boolean).map(x => x.value || defaultValue) || [];
```

— Коротко, красиво, опасно.

Лучше:

```js
const items = data?.items ?? [];

const result = items
  .filter(item => item != null)
  .map(item => item.value ?? defaultValue);
```

— Длиннее, зато видно, что происходит.

Мальчик долго смотрел на эти две версии.

— В первой версии я как будто еду на ржавом велосипеде без тормозов, потому что дорога пока прямая. Во второй — иду пешком, зато вижу ямы.

— Иногда можно ехать на велосипеде. Но сначала надо знать, где тормоза.
